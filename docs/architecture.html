High-Throughput Sync & Migration Suite

Bản vẽ đặc tả kiến trúc tổng thể, mô hình luồng dữ liệu, sự tương tác giữa main-server, search_sync_worker và các tầng lưu trữ dữ liệu (MongoDB, PostgreSQL, Redis, Elasticsearch).

Tổng quan

Tổng quan về Kiến trúc Hệ thống

Hệ thống Main là một nền tảng Live-streaming và mạng xã hội chịu tải lớn (High CCU). Kiến trúc của hệ thống được tối ưu hóa để giải quyết đồng thời hai bài toán nghiệp vụ phức tạp:

  • Đồng bộ Tìm kiếm Toàn văn (Search Indexing Pipeline): Chuyển dịch và cập nhật thời gian thực dữ liệu tin nhắn (messages) và người dùng (users) từ MongoDB sang Elasticsearch phục vụ tính năng tìm kiếm siêu tốc.
  • Di chuyển Cơ sở Dữ liệu Không gián đoạn (Zero-Downtime Database Migration): Từng bước dịch chuyển cấu phần mối quan hệ người dùng (user_relation) từ MongoDB sang PostgreSQL để đảm bảo tính toàn vẹn dữ liệu.
Nguyên lý thiết kế cốt lõi:
Tách biệt tuyệt đối giữa luồng xử lý API giao dịch trực tuyến (Online Transaction Flow) đảm nhiệm bởi main-server và luồng xử lý đồng bộ bất đồng bộ (Offline/Asynchronous Processing Pipeline) đảm nhiệm bởi search_sync_worker.

Cấu phần 1

1. main-server (Backend Application Layer)

Là máy chủ API chính nhận lưu lượng truy cập từ Client. Tận dụng kiến trúc linh hoạt để thực hiện quá trình chuyển đổi database an toàn thông qua hai kỹ thuật:

Dual-Write (Ghi kép đồng thời)

Khi người dùng thực hiện một hành động ghi mới (ví dụ: tạo mối quan hệ follow giữa 2 user), main-server sẽ thực thi:

  1. Ghi đồng bộ vào MongoDB (được chọn là Primary DB để đảm bảo luồng API cũ hoạt động 100% bình thường).
  2. Bọc luồng ghi vào PostgreSQL (Secondary DB) trong một tác vụ bất đồng bộ độc lập (Background thread hoặc Event Listener), bắt sạch exception để lỗi bên Postgres không làm hỏng request chính của user.

Shadow-Read (Đọc bóng đối chiếu)

Để kiểm nghiệm chất lượng dữ liệu của PostgreSQL và Elasticsearch trước khi chính thức đưa vào sử dụng:

  1. Ứng dụng đọc dữ liệu trực tiếp từ DB tin cậy cũ (MongoDB).
  2. Kích hoạt một Shadow Thread ngầm đọc bản ghi tương ứng bên DB mới (Postgres hoặc Elasticsearch).
  3. Sử dụng thuật toán so khớp chéo (Deep-Compare) để phát hiện chênh lệch dữ liệu (Mismatch) và ghi log giám sát.

Dynamic Feature Flags (Cấu hình chuyển đổi nóng)

Các chế độ đọc ghi được điều phối linh hoạt thông qua cấu hình cờ (Flags) lưu tại Redis, cho phép chuyển mạch hệ thống mà không cần restart server:

Tên chế độ (Mode) Hành vi Đọc (Read Path) Hành vi Ghi (Write Path)
MONGO_ONLY Chỉ đọc từ MongoDB Chỉ ghi vào MongoDB
DUAL_WRITE & SHADOW_READ Đọc MongoDB + Đọc ngầm Postgres/ES so khớp Ghi MongoDB + Ghi ngầm Postgres/ES (try-catch)
POSTGRES_PRIMARY Đọc Postgres chính + Đọc ngầm Mongo đối chiếu Ghi Postgres chính + Ghi ngầm Mongo
POSTGRES_ONLY Chỉ đọc từ PostgreSQL Chỉ ghi vào PostgreSQL
Cấu phần 2

2. search_sync_worker (Data Sync Pipeline)

Là dịch vụ chạy nền (chạy bằng Python hoặc Node.js) chịu trách nhiệm đồng bộ dữ liệu cũ và dữ liệu mới sinh từ MongoDB sang Elasticsearch Index. Pipeline được thiết kế làm hai pha rõ rệt:

Pha 1: Initial Load (Backfile / Đồng bộ ban đầu)

Thực hiện đồng bộ toàn bộ dữ liệu lịch sử từ MongoDB sang Elasticsearch trước khi bật stream:

  • Phân trang Keyset Pagination: Sử dụng truy vấn {_id: {'$gt': last_id}} kết hợp sort tăng dần theo _id. Tốc độ quét luôn đạt mức tối ưu $O(\log N)$ nhờ chỉ mục index mặc định trên khóa chính.
  • Tối ưu hóa Write Index Settings: Trước khi chạy, worker thay đổi settings của ES index thành refresh_interval: -1 (tắt sinh segment nhỏ) và number_of_replicas: 0 (tắt nhân bản) giúp tăng tốc độ bulk insert lên gấp 3 lần. Cấu hình được khôi phục về chế độ sản xuất an toàn khi bulk hoàn tất.
  • Parallel Bulk Write: Gom dữ liệu thành từng batch 1000 bản ghi, gửi đa luồng song song thông qua cơ chế parallel_bulk của Elasticsearch Client.

Pha 2: Watch Change Stream (Changestream / Đồng bộ thời gian thực)

Sau khi hoàn thành Backfile, worker chuyển mạch sang chế độ CDC (Change Data Capture) để đồng bộ thời gian thực:

  • Lắng nghe MongoDB Oplog: Sử dụng hàm db.collection.watch() để lắng nghe toàn bộ các sự kiện thay đổi (insert, update, replace, delete) trên collection MongoDB.
  • Bộ đệm gom lô (Buffer & Flush): Để bảo vệ Elasticsearch khỏi quá tải do ghi lẻ tẻ, worker gom các sự kiện vào bộ đệm và chỉ flush hàng loạt khi đạt số lượng SYNC_BATCH_SIZE (1000 docs) hoặc sau khoảng thời gian SYNC_POLL_INTERVAL (2 giây).
  • Quản lý trạng thái (State & Checkpointing) trên Redis:
    • Pha Backfile: Lưu lại last_id (BSON ObjectId) của bản ghi cuối được xử lý.
    • Pha Changestream: Lưu lại clusterTime (BSON Timestamp) của sự kiện change stream cuối được flush thành công.
  • Tự động khôi phục lỗi Oplog Rollover: Khi worker bị ngắt kết nối quá lâu khiến Resume Token bị lỗi thời do Oplog cũ bị ghi đè, hệ thống tự động bắt lỗi (ví dụ lỗi code 136 hoặc invalid resume token), tự động xóa checkpoint trên Redis và chạy lại tiến trình Backfile tự động để tái thiết lập tính nhất quán của dữ liệu.
Cấu phần 3

3. Vai trò của các Tầng lưu trữ (Database Roles)

Mỗi cơ sở dữ liệu đóng một vai trò chuyên biệt, phối hợp tạo nên tính chịu tải cao và bền bỉ cho hệ thống:

Database Mô hình dữ liệu Vai trò trong hệ thống Lý do lựa chọn
MongoDB Document Store (NoSQL) Lưu trữ giao dịch gốc (Primary Source of Truth) ban đầu của messages và users. Tốc độ ghi chép dữ liệu dạng JSON thô cực nhanh, phù hợp cho dữ liệu live chat.
PostgreSQL Relational (SQL) Nguồn dữ liệu giao dịch sạch đích cho phân hệ user_relation. Đảm bảo ràng buộc toàn vẹn khóa ngoại (Foreign Keys), tối ưu hóa các câu truy vấn quan hệ phức tạp (Mutual Friends).
Elasticsearch Search Engine (Inverted Index) Chỉ mục tìm kiếm toàn văn. Đọc chịu tải chính cho tính năng tìm kiếm user và message. Tìm kiếm văn bản siêu tốc, hỗ trợ phân tích ngôn ngữ tự nhiên, tách tải đọc tìm kiếm ra khỏi DB chính.
Redis RAM In-Memory Key-Value Lưu trữ trạng thái Checkpoint đồng bộ của worker + Lưu trữ Feature Flags chuyển mạch cấu hình. Độ trễ truy cập cực thấp (micro-seconds), hỗ trợ cấu hình chuyển mạch động tức thì.
Quy trình vận hành

4. Quy trình di chuyển dữ liệu (Migration Workflow)

Chiến dịch di chuyển phân hệ user_relation từ MongoDB sang PostgreSQL được thực hiện tuần tự qua 5 bước sau để đảm bảo an toàn tuyệt đối:

BƯỚC 1: Kích hoạt Dual-Write (Ghi kép)

Bật flag WRITE_MODE = DUAL_WRITE_MONGO_PRIMARY. Mọi quan hệ follow/block mới sinh ra sẽ được ghi đồng thời vào MongoDB và PostgreSQL ngầm. Postgres hoạt động dưới dạng non-blocking task để bảo vệ luồng API chính.

BƯỚC 2: Di chuyển dữ liệu lịch sử (Historical Batch Migration)

Khởi chạy Batch Job quét toàn bộ quan hệ cũ trên MongoDB ghi sang PostgreSQL. Sử dụng câu lệnh ghi đè Postgres ON CONFLICT ... DO UPDATE để tránh lỗi trùng lặp khóa với các bản ghi được ghi kép ở Bước 1.

BƯỚC 3: Kích hoạt Shadow-Read & Đối soát ngầm (Data Verification)

Bật flag READ_MODE = SHADOW_READ_COMPARE. Hệ thống đọc từ MongoDB và đối chiếu ngầm với PostgreSQL. Dev theo dõi log mismatch, chạy các tiến trình Reconciliation sửa lỗi tự động cho đến khi tỷ lệ khớp đạt 100%.

BƯỚC 4: Hoán đổi nguồn dữ liệu chính (Cutover)

Bật flag WRITE_MODE = DUAL_WRITE_PG_PRIMARY và READ_MODE = PG_PRIMARY_SHADOW_MONGO. PostgreSQL chính thức trở thành nguồn dữ liệu chính phục vụ nghiệp vụ đọc/ghi. MongoDB trở thành DB phụ ghi ngầm để dự phòng.

BƯỚC 5: Ngắt kết nối hoàn toàn (Deprecation)

Sau thời gian theo dõi hệ thống chạy ổn định trên Postgres, bật flag WRITE_MODE = PG_ONLY và READ_MODE = PG_ONLY. Ngắt kết nối và giải phóng hoàn toàn MongoDB đối với phân hệ user_relation.
