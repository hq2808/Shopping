1. MongoDB Aggregation Pipeline trong Spring Data MongoDB

MongoTemplate

Trong MongoDB, Aggregation Pipeline là công cụ mạnh mẽ để xử lý, biến đổi và thống kê dữ liệu qua nhiều giai đoạn (Stages) nối tiếp. Đối với lập trình viên Senior, việc nắm vững cách phối hợp các stage như $match, $lookup, $unwind, $group, $project và tối ưu hóa hiệu năng của chúng trong Java là bắt buộc.

Lưu ý tối ưu hóa hiệu năng (Performance Tuning):

  • Luôn đặt stage $match và $sort lên đầu tiên của pipeline để tận dụng tối đa Index. Nếu đặt sau $project hay $lookup, MongoDB sẽ buộc phải quét toàn bộ Collection (Collscan).
  • Mặc định, mỗi Stage trong Aggregation bị giới hạn bộ nhớ RAM tối đa **100MB**. Nếu tập dữ liệu lớn vượt ngưỡng, truy vấn sẽ bị crash. Cần cấu hình allowDiskUse(true) để cho phép ghi dữ liệu tạm ra ổ đĩa nếu cần thiết.

OrderAggregationRepositoryImpl.java (Production Ready với Join $lookup) Java 
@Repository
public class OrderAggregationRepositoryImpl {
    private final MongoTemplate mongoTemplate;

    @Autowired
    public OrderAggregationRepositoryImpl(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    public List<UserOrderRevenueDto> getUserOrderRevenueReport(String status, double minSpent) {
        // Stage 1: Match - Lọc các order thành công bằng index
        MatchOperation matchOrders = Aggregation.match(
            Criteria.where("status").is(status).and("deletedAt").exists(false)
        );

        // Stage 2: Lookup - Thực hiện Join nối bảng với Collection 'users'
        LookupOperation lookupUser = LookupOperation.newLookup()
            .from("users")                  // Collection cần join
            .localField("userId")          // Khóa ngoại bên Collection orders
            .foreignField("_id")           // Khóa chính bên Collection users
            .as("userDetails");            // Tên mảng kết quả sau khi join

        // Stage 3: Unwind - Làm phẳng mảng kết quả join thành object đơn lẻ
        UnwindOperation unwindUser = Aggregation.unwind("userDetails");

        // Stage 4: Group - Gom nhóm theo User, tính tổng doanh thu và đếm số đơn hàng
        GroupOperation groupUser = Aggregation.group("userDetails._id")
            .first("userDetails.email").as("userEmail")
            .first("userDetails.name").as("userName")
            .sum("price").as("totalSpent")
            .count().as("totalOrders");

        // Stage 5: Match (Post-group) - Lọc tiếp các User tiêu dùng vượt ngưỡng yêu cầu
        MatchOperation filterRevenue = Aggregation.match(Criteria.where("totalSpent").gte(minSpent));

        // Stage 6: Project - Định hình cấu trúc JSON trả về cho Client
        ProjectionOperation projectFields = Aggregation.project()
            .and("_id").as("userId")
            .andInclude("userEmail", "userName", "totalSpent", "totalOrders")
            .andExclude("_id"); // Ẩn trường ID gốc

        // Stage 7: Sort - Sắp xếp theo tổng tiền tiêu dùng giảm dần
        SortOperation sortByRevenue = Aggregation.sort(Sort.Direction.DESC, "totalSpent");

        // Khởi tạo Aggregation với tùy chọn allowDiskUse để chống lỗi tràn RAM (100MB limit)
        Aggregation aggregation = Aggregation.newAggregation(
            matchOrders, lookupUser, unwindUser, groupUser, filterRevenue, projectFields, sortByRevenue
        ).withOptions(Aggregation.newAggregationOptions().allowDiskUse(true).build());

        AggregationResults<UserOrderRevenueDto> results = 
            mongoTemplate.aggregate(aggregation, "orders", UserOrderRevenueDto.class);
        return results.getMappedResults();
    }
}

2. Giải quyết lỗi N+1 Query trong Spring Data JPA

Hibernate Optimization

Sử dụng truy vấn nạp dữ liệu chủ động (Eager Fetching) để tránh việc sinh ra hàng loạt truy vấn con khi duyệt danh sách:

UserRepository.java Java 
public interface UserRepository extends JpaRepository<User, Long> {
    // Cách 1: Sử dụng JOIN FETCH trong JPQL truy vấn
    @Query("SELECT u FROM User u LEFT JOIN FETCH u.orders WHERE u.email = :email")
    Optional<User> findByEmailWithOrdersFetch(String email);

    // Cách 2: Sử dụng EntityGraph động của JPA 2.1
    @EntityGraph(attributePaths = {"orders", "roles"})
    List<User> findAll();
}

3. Bộ câu hỏi phỏng vấn Senior (Q&A)

Làm cách nào cấu hình Giao dịch đa tài liệu (Multi-document Transaction) trong MongoDB sử dụng Spring Boot? Những lưu ý về hiệu năng là gì?
Để sử dụng @Transactional với MongoDB, ta phải đăng ký Bean MongoTransactionManager trong file cấu hình. Đồng thời, MongoDB yêu cầu database phải đang chạy dưới dạng Replica Set hoặc Sharded Cluster.
Lưu ý hiệu năng: Giao dịch đa tài liệu khóa các document liên quan và sinh ra chi phí đồng bộ lớn. Chỉ sử dụng khi thực sự cần tính toàn vẹn dữ liệu tuyệt đối (Ví dụ: Khấu trừ tiền ví điện tử để mua vé livestream). Tránh các giao dịch chạy quá lâu (hơn 60 giây mặc định) vì sẽ làm nghẽn bộ nhớ đệm có dây (WiredTiger cache).
Sự khác nhau giữa First-level Cache và Second-level Cache trong Hibernate? Khi nào nên cấu hình L2 Cache?
First-level Cache: Gắn liền với Session (hoặc EntityManager), là cache mặc định và hoạt động ở mức luồng (transaction scope). Dữ liệu tự động giải phóng khi transaction kết thúc.
Second-level Cache: Hoạt động ở mức SessionFactory (application scope), chia sẻ giữa tất cả các session. Cần tích hợp thư viện ngoài (như Ehcache, Redis).
Khi nào dùng L2 Cache: Thích hợp cho các bảng dữ liệu ít biến động nhưng đọc rất nhiều (như bảng Config hệ thống, danh mục quốc gia). Không dùng cho các bảng ghi liên tục vì chi phí đồng bộ cache (eviction/invalidation) sẽ làm giảm hiệu năng hệ thống.
Khi nào ta nên bỏ qua cơ chế mapping của JPA/MongoEngine để chuyển sang viết truy vấn Native SQL/Raw Document?
Khi cần thực hiện các thao tác ghi dữ liệu lớn hàng loạt (Bulk Write) hoặc tải các tập dữ liệu khổng lồ chỉ đọc để kết xuất báo cáo. Quá trình ánh xạ (Mapping) hàng triệu bản ghi từ DB sang Object thực thể Java tiêu tốn rất nhiều tài nguyên CPU và bộ nhớ JVM (sinh ra rác). Sử dụng truy vấn Native SQL trả về DTO trực tiếp hoặc thao tác trực tiếp với Driver (ví dụ Pymongo hoặc mongoTemplate.getCollection()) để bypass cơ chế quản lý vòng đời (Persistence Context) của ORM, tăng tốc độ xử lý gấp nhiều lần.

4. Cấu hình Index MongoDB chuyên sâu

Hiểu cơ chế hoạt động của Compound Index, luật ưu tiên tiền tố (Prefix Rule). Thiết kế Index cho các bài toán truy vấn lồng nhau (Embedded Document Index).

5. Chiến lược khóa dữ liệu (Pessimistic Locking vs Optimistic Locking)

Tránh race condition khi cập nhật số dư ví điện tử hoặc số lượng vé livestream. Cách triển khai `@Version` trong JPA và toán tử nguyên tử `$inc` trong MongoDB.

6. Tối ưu hóa Database Connection Pool

Cấu hình tối ưu thư viện HikariCP (Connection Timeout, Maximum Pool Size) cho các ứng dụng Spring Boot chịu tải cao.

7. Hibernate Lazy Initialization Exception

Nguyên nhân xảy ra lỗi khi cố truy cập thuộc tính Lazy load ngoài phạm vi Transaction (Session đã đóng) và cách khắc phục triệt để bằng DTO Projections.
