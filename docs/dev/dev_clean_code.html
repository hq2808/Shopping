1. Refactoring & Code Smells phổ biến

Ở cấp độ Senior, việc nhận diện và cấu trúc lại các đoạn mã phức tạp (Refactoring) là vô cùng quan trọng. Dưới đây là cách ánh xạ các lỗi viết code phổ biến sang Java:

  • Bloaters (Mã phình to):
    • Long Method (> 15 lines): Khắc phục bằng cách sử dụng Extract Method, hoặc chuyển đổi biến tạm thành truy vấn (Replace Temp with Query).
    • Long Parameter List: Gộp các tham số liên quan thành một đối tượng chứa cấu hình (Introduce Parameter Object hoặc áp dụng Builder Pattern trong Java).
  • Object-Orientation Abusers:
    • Switch Statements: Thay thế các khối switch-case phức tạp bằng tính đa hình (Replace Conditional with Polymorphism) kết hợp với Strategy Pattern.
  • Change Preventers (Ngăn cản thay đổi):
    • Shotgun Surgery: Khi thay đổi một tính năng yêu cầu sửa đổi quá nhiều Class nhỏ lẻ khác nhau. Lỗi này vi phạm nghiêm trọng nguyên lý Single Responsibility Principle (SRP).

2. Thực tế Java Refactoring: Thay thế Switch bằng Strategy + Spring Map Injection

Trong các dự án lớn, việc lạm dụng khối switch-case (hoặc một chuỗi if-else if kéo dài) để phân loại xử lý logic theo điều kiện là lỗi thiết kế cực kỳ phổ biến (Switch Statements smell). Nó vi phạm nguyên lý Open/Closed Principle (OCP) của SOLID vì mỗi khi thêm một loại nghiệp vụ mới, ta buộc phải mở mã nguồn cũ của lớp điều phối ra để chỉnh sửa.

Giải pháp chuẩn Senior: Sử dụng Strategy Pattern kết hợp cơ chế Dependency Injection mạnh mẽ của Spring Framework để tự động nạp các Strategy Beans vào một Map lưu trữ tập trung.

PaymentStrategy.java & PaymentService.java (Production Ready) Java 
import java.math.BigDecimal;

/**
 * Strategy Interface định nghĩa hành vi chung cho tất cả các cổng thanh toán.
 */
public interface PaymentStrategy {
    void processPayment(String orderId, BigDecimal amount);
    String getPaymentType(); // Trả về định danh duy nhất của Strategy
}

/**
 * Implementation xử lý thanh toán qua thẻ tín dụng.
 */
@Component
public class CreditCardPayment implements PaymentStrategy {
    @Override
    void processPayment(String orderId, BigDecimal amount) {
        // Logic kết nối cổng Visa/MasterCard
    }
    
    @Override
    public String getPaymentType() {
        return "CREDIT_CARD";
    }
}

/**
 * Implementation xử lý thanh toán qua Apple Pay.
 */
@Component
public class ApplePayPayment implements PaymentStrategy {
    @Override
    void processPayment(String orderId, BigDecimal amount) {
        // Logic mã hóa mã token Apple Pay
    }
    
    @Override
    public String getPaymentType() {
        return "APPLE_PAY";
    }
}

/**
 * Service điều phối (Context) sử dụng cơ chế Constructor Injection 
 * để tự động gom toàn bộ các Bean triển khai PaymentStrategy.
 */
@Service
public class PaymentService {
    // Immutable Map làm registry lưu trữ các Strategy
    private final Map<String, PaymentStrategy> paymentRegistry;

    /**
     * Spring sẽ tự động quét classpath, tìm kiếm tất cả các class được đánh dấu
     * @Component/Service triển khai PaymentStrategy và nạp vào List.
     */
    @Autowired
    public PaymentService(List<PaymentStrategy> strategies) {
        Map<String, PaymentStrategy> tempMap = new HashMap<>();
        strategies.forEach(strategy -> {
            PaymentStrategy duplicate = tempMap.put(strategy.getPaymentType(), strategy);
            if (duplicate != null) {
                throw new IllegalStateException("Trùng lặp cổng thanh toán: " + strategy.getPaymentType());
            }
        });
        // Chuyển sang Map không thể sửa đổi (Unmodifiable) sau khi khởi động để đảm bảo Thread-safe
        this.paymentRegistry = Collections.unmodifiableMap(tempMap);
    }

    public void executePayment(String paymentType, String orderId, BigDecimal amount) {
        PaymentStrategy strategy = paymentRegistry.get(paymentType);
        if (strategy == null) {
            throw new UnsupportedOperationException("Không tìm thấy dịch vụ thanh toán: " + paymentType);
        }
        strategy.processPayment(orderId, amount);
    }
}
💡 Tại sao giải pháp này được các System Architects ưa thích?
  • Hoàn toàn tuân thủ OCP: Khi dự án tích hợp thêm một cổng thanh toán mới (ví dụ: Google Pay, Momo), lập trình viên chỉ cần tạo một class mới triển khai PaymentStrategy và đánh dấu @Component. Lớp PaymentService và các đoạn code cũ không cần sửa đổi hay re-deploy, giảm thiểu tối đa rủi ro gây lỗi hồi quy (regression bugs).
  • Dễ dàng viết Unit Test: Bạn có thể dễ dàng mock từng PaymentStrategy riêng lẻ mà không lo lắng về các nhánh điều kiện ẩn bên trong câu lệnh switch lớn.
  • Tối ưu hóa Thread-safe: Bản đồ Registry được khởi tạo một lần duy nhất lúc khởi động hệ thống và chuyển đổi sang dạng unmodifiableMap, giúp chống race condition hoàn hảo khi hàng ngàn luồng truy cập cùng lúc.

3. Bộ Câu hỏi Phỏng vấn Senior (Q&A)

Làm thế nào để tránh tình trạng "Over-engineering" (Thiết kế quá mức cần thiết) khi refactoring code?
Cần tuân thủ nguyên lý YAGNI (You Aren't Gonna Need It) và KISS (Keep It Simple, Stupid). Không tạo interface trừ khi có ít nhất 2 implementation thực tế hoặc cần thực hiện Mocking trong Unit Test. Chỉ refactor khi đoạn mã đó cần thay đổi thường xuyên (vi phạm OCP - Open/Closed Principle) hoặc gặp lỗi hiệu năng thực tế.
Builder Pattern giải quyết bài toán gì của Constructor trong Java, và làm sao đảm bảo tính Thread-safe?
Builder Pattern giải quyết bài toán "Telescoping Constructor" (quá nhiều tham số khởi tạo chồng chéo). Nó giúp tăng tính tường minh và cho phép kiểm soát tính hợp lệ của dữ liệu trước khi build. Để đảm bảo an toàn luồng và tính bất biến (immutability), tất cả các trường trong thực thể chính nên được khai báo là private final và không có phương thức setter.
Sự khác nhau giữa Composition và Inheritance (Kế thừa)? Khi nào nên ưu tiên Composition?
Inheritance tạo mối quan hệ chặt chẽ "IS-A", dễ dẫn đến vi phạm tính đóng gói nếu lớp cha thay đổi (Refused Bequest smell). Composition tạo mối quan hệ lỏng "HAS-A", cho phép thay đổi hành vi động lúc runtime (ví dụ qua Strategy Pattern). Chuẩn Senior luôn ưu tiên Composition hơn Inheritance ("Favor composition over inheritance") để giữ hệ thống linh hoạt và giảm thiểu rủi ro lan truyền lỗi.

4. Nguyên lý SOLID chuyên sâu

Phân tích các vi phạm thực tế đối với nguyên lý Liskov Substitution Principle (LSP) và Interface Segregation Principle (ISP) trong các dự án Spring Boot lớn.

5. Các công cụ kiểm tra tĩnh (Static Analysis Tools)

Tìm hiểu cách tích hợp SonarQube, SpotBugs, ArchUnit (để kiểm tra vi phạm kiến trúc kiến thức) và Checkstyle vào CI/CD pipeline cho dự án Java.

6. Tối ưu hóa hiệu năng Clean Code

Tránh các lỗi tạo rác bộ nhớ (Memory Leak), lạm dụng String concatenation thay vì StringBuilder, cách sử dụng hợp lý các Collection trong luồng chạy đa luồng.

7. Viết Pull Request & Code Review chuẩn mực

Cách phân rã các tính năng lớn thành các Stacked PRs trong Java, sử dụng Git pre-commit hooks cho auto-formatter (như Google Java Format).
