HTTP & Spring Security chuyên sâu

Khảo sát sâu cơ chế mạng, các bộ lọc Servlet trong Spring Boot và xây dựng hệ thống phân quyền JWT stateless an toàn.

📈 Tiền độ ôn tập chủ đề

1. So sánh HTTP/1.1 vs HTTP/2 vs HTTP/3 trong Spring Boot

Hiểu rõ cách tối ưu hóa giao thức mạng ở tầng Web Server (Tomcat/Undertow) của Spring Boot:

  • HTTP/2: Giải quyết lỗi Head-of-Line Blocking ở mức HTTP bằng cơ chế Multiplexing. Trong Spring Boot, có thể cấu hình HTTP/2 dễ dàng qua file application.properties bằng việc bật SSL (HTTP/2 yêu cầu TLS).
  • HTTP/3: Sử dụng giao thức QUIC chạy trên UDP. Hiện tại, Spring Boot (thông qua Project Reactor Netty) đã có các thử nghiệm tích hợp HTTP/3 giúp tăng tốc thiết lập kết nối (0-RTT) và chống rớt gói tin cực tốt khi chuyển đổi mạng (nhờ Connection ID).

2. Bộ lọc ETag Caching với ShallowEtagHeaderFilter

Tiết kiệm băng thông mạng bằng cách gửi mã phản hồi 304 Not Modified khi nội dung tài nguyên không thay đổi:

WebMvcConfig.java Java 
@Configuration
public class WebMvcConfig {
    @Bean
    public FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
        FilterRegistrationBean<ShallowEtagHeaderFilter> registration = 
            new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
        registration.addUrlPatterns("/api/products/*", "/api/assets/*");
        return registration;
    }
}

3. Bộ lọc xác thực Stateless JWT trong Spring Security

Trong kiến trúc microservice hoặc RESTful API stateless, Spring Security không duy trì HttpSession của người dùng. Thay vào đó, mỗi request phải tự đính kèm JWT trong header. Lớp bộ lọc kế thừa từ OncePerRequestFilter đảm bảo kiểm tra token này một lần duy nhất cho mỗi yêu cầu.

Lưu ý cấp độ Senior: Bộ lọc JWT chạy trước Spring Security filter chain. Nếu xảy ra lỗi (token hết hạn, chữ ký sai), việc ném Exception trực tiếp trong Filter sẽ làm sập request và trả về lỗi 500 mặc định vì nó nằm ngoài tầm kiểm soát của ControllerAdvice. Code chuẩn cần bắt Exception của JWT và ghi dữ liệu lỗi JSON chuẩn (401 Unauthorized) trực tiếp vào HttpServletResponse.

JwtAuthenticationFilter.java (Chuẩn Production) Java 
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtProvider jwtProvider;
    private final UserDetailsService userDetailsService;
    private final ObjectMapper objectMapper = new ObjectMapper();

    @Autowired
    public JwtAuthenticationFilter(JwtProvider jwtProvider, UserDetailsService userDetailsService) {
        this.jwtProvider = jwtProvider;
        this.userDetailsService = userDetailsService;
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        // Bỏ qua các API public không cần xác thực để tránh giải mã token vô ích
        String path = request.getServletPath();
        return path.startsWith("/api/auth/") || path.startsWith("/public/");
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                    HttpServletResponse response, 
                                    FilterChain filterChain) throws ServletException, IOException {
        String token = resolveToken(request);
        if (token == null) {
            filterChain.doFilter(request, response);
            return;
        }

        try {
            if (jwtProvider.validateToken(token)) {
                String username = jwtProvider.getUsernameFromToken(token);
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);
                
                UsernamePasswordAuthenticationToken authentication = 
                    new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
                authentication.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                
                // Ghi nhận thông tin xác thực vào SecurityContext của Spring
                SecurityContextHolder.getContext().setAuthentication(authentication);
            }
            filterChain.doFilter(request, response);
        } catch (ExpiredJwtException ex) {
            handleException(response, "Token đã hết hạn sử dụng");
        } catch (SignatureException | MalformedJwtException ex) {
            handleException(response, "Chữ ký Token không hợp lệ");
        } catch (Exception ex) {
            handleException(response, "Lỗi xác thực Token hệ thống");
        }
    }

    private String resolveToken(HttpServletRequest request) {
        String bearerToken = request.getHeader("Authorization");
        if (StringUtils.hasText(bearerToken) && bearerToken.startsWith("Bearer ")) {
            return bearerToken.substring(7);
        }
        return null;
    }

    private void handleException(HttpServletResponse response, String message) throws IOException {
        SecurityContextHolder.clearContext(); // Xóa bỏ context nếu có lỗi xảy ra
        response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
        response.setContentType("application/json; charset=utf-8");
        
        Map<String, Object> errorDetails = Map.of(
            "timestamp", Instant.now().toString(),
            "status", HttpServletResponse.SC_UNAUTHORIZED,
            "error", "Unauthorized",
            "message", message
        );
        response.getWriter().write(objectMapper.writeValueAsString(errorDetails));
    }
}

4. Bộ câu hỏi phỏng vấn Senior (Q&A)

Lọc ShallowEtagHeaderFilter hoạt động thế nào dưới nền? Nó có giúp giảm tải cho database không?
ShallowEtagHeaderFilter hoạt động bằng cách ghi đè OutputStream của Response và tính toán mã băm MD5 dựa trên toàn bộ body nhận được. Nó sẽ đối chiếu mã MD5 này với header If-None-Match từ Client. Nếu trùng, nó xóa sạch Body và trả về mã 304.
Chú ý: Bộ lọc này KHÔNG giúp giảm tải cho database, vì Controller vẫn xử lý bình thường và query DB bình thường. Để giảm tải DB, ta phải dùng cơ chế Deep ETag (truy vấn nhanh hash/last-modified từ DB trước) hoặc ứng dụng Cache cấp độ Repository.
Vì sao nên chọn OncePerRequestFilter thay vì bộ lọc Filter thông thường trong Spring Security?
Mỗi Servlet Filter thông thường có thể bị gọi nhiều lần cho cùng một request trong các kịch bản Forwarding (Chuyển tiếp trang nội bộ) hoặc xử lý lỗi bất đồng bộ. OncePerRequestFilter đảm bảo logic xác thực token (như giải mã JWT, truy vấn UserDetails) chỉ chạy đúng **một lần duy nhất** cho mỗi request gửi lên, tối ưu tài nguyên CPU và kết nối.
Làm thế nào để phòng chống tấn công CSRF khi xây dựng hệ thống Stateless REST API sử dụng JWT?
Nếu JWT được lưu trong Cookie (với cờ HttpOnly và Secure để chống XSS), ứng dụng sẽ bị phơi nhiễm trước tấn công CSRF. Để phòng chống, ta có thể triển khai cơ chế Double Submit Cookie, yêu cầu Client đính kèm một token CSRF ngẫu nhiên trong Custom HTTP Header (ví dụ X-XSRF-TOKEN). Vì các trang web độc hại không thể đọc/ghi cookie của domain khác do chính sách Same-Origin Policy, chúng không thể đính kèm Header này. Nếu JWT lưu trong LocalStorage, ta không lo bị tấn công CSRF, nhưng phải có các biện pháp kiểm soát chống tấn công XSS nghiêm ngặt (như cấu hình Content Security Policy - CSP).

5. Luồng xoay vòng Refresh Token (Token Rotation) an toàn

Cách quản lý Refresh Token an toàn: lưu trữ dạng băm một chiều trong Redis, thiết lập thời gian hết hạn TTL tương ứng và hủy toàn bộ các token cũ khi phát hiện có hành vi đăng nhập/sử dụng lại token đáng ngờ.

6. Quản lý cấu hình CORS nâng cao

Tránh các lỗi bảo mật cấu hình CORS tùy tiện (*) khi API yêu cầu chứng thực credentials. Cách triển khai CorsConfigurationSource chuẩn chỉ.

7. Cấu hình HSTS & bảo mật Header trong Spring Security

Bảo vệ ứng dụng chống tấn công Man-in-the-middle bằng cách ép trình duyệt chỉ kết nối HTTPS thông qua cấu hình Strict-Transport-Security (HSTS) và các header bảo mật (X-Frame-Options, CSP).

8. RESTful API error handling toàn cục

Xây dựng hệ thống xử lý lỗi đồng bộ bằng cách sử dụng @ControllerAdvice kết hợp @ExceptionHandler để ẩn đi các lỗi StackTrace nhạy cảm của hệ thống trước Client.
