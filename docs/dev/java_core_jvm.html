1. Hợp đồng equals()/hashCode() & Kiến trúc HashMap/ConcurrentHashMap nâng cao

Java Collections

1.1. Hợp đồng bất biến (The Contract)

Trong Java, sự nhất quán giữa equals() và hashCode() là nền tảng để các Collections dựa trên mã băm hoạt động đúng đắn. Quy tắc cốt lõi:

  • Nếu obj1.equals(obj2) == true, thì bắt buộc obj1.hashCode() == obj2.hashCode().
  • Nếu hai đối tượng có cùng giá trị băm (hashCode), chúng không nhất thiết phải bằng nhau qua equals(). Hiện tượng này gọi là Va chạm mã băm (Hash Collision).

1.2. Cơ chế Treeify (Đỏ-Đen hóa) của HashMap

HashMap lưu trữ dữ liệu dưới dạng một mảng các node (gọi là các buckets). Khi xảy ra va chạm băm:

  • Java 7: Node mới được chèn vào đầu một danh sách liên kết đơn (Linked List). Khi độ dài danh sách quá lớn, thao tác tìm kiếm bị suy giảm hiệu năng xuống O(N).
  • Java 8+: Khi số lượng phần tử trong một bucket vượt qua ngưỡng TREEIFY_THRESHOLD = 8 và đồng thời tổng dung lượng mảng bucket tối thiểu phải đạt MIN_TREEIFY_CAPACITY = 64, Java sẽ tự động chuyển đổi Linked List này thành một Cây Đỏ-Đen (Red-Black Tree).
  • Lúc này, độ phức tạp tìm kiếm được giảm từ O(N) xuống O(log N). Nếu số lượng phần tử trong bucket giảm xuống còn UNTREEIFY_THRESHOLD = 6 (do bị xóa hoặc rehash), cấu trúc cây sẽ quay trở lại dạng Linked List để tiết kiệm chi phí vận hành.

1.3. Tiến hóa của ConcurrentHashMap: Java 7 Segment Locks vs Java 8+ CAS & Synchronized

Đối với hệ thống đa luồng, HashMap thông thường sẽ bị lỗi vòng lặp vô hạn (Infinite Loop) hoặc mất dữ liệu khi rehash đồng thời. ConcurrentHashMap giải quyết vấn đề này qua hai thế hệ kiến trúc:

Đặc tính Java 7 (Segment Locking) Java 8+ (CAS & Node Locking)
Cơ chế khóa Sử dụng một mảng các Segment độc lập, mỗi segment kế thừa ReentrantLock. Khóa phân đoạn riêng biệt. Không dùng Segment. Dùng cơ chế khóa nguyên tử CAS (Compare-And-Swap) kết hợp synchronized trực tiếp trên Node đầu của bucket.
Độ mịn khóa (Concurrency Level) Độ mịn cố định (mặc định 16). Muốn tăng phải cấu hình thủ công lúc khởi tạo. Luồng ghi đụng segment sẽ bị chặn. Độ mịn cực cao bằng số bucket thực tế. Chỉ khóa duy nhất bucket xảy ra tranh chấp ghi, các bucket khác chạy hoàn toàn độc lập.
Hiệu năng đọc Không bị khóa nhờ biến volatile. Không bị khóa nhờ đọc qua volatile Node, hiệu năng đọc cực đại.

Tại sao ConcurrentHashMap không cho phép Key hoặc Value bằng Null?

Đây là một thiết kế có chủ đích của tác giả Doug Lea để tránh hiện tượng mập mờ (ambiguity) trong môi trường phân tán/đa luồng. Nếu cho phép trả về null, khi ta gọi map.get(key) nhận được null, ta không thể biết chắc chắn:

  1. Key đó không tồn tại trong map.
  2. Key đó có tồn tại nhưng giá trị của nó được gán bằng null.

Trong Map thông thường (như HashMap), ta có thể kiểm tra bằng map.containsKey(key). Tuy nhiên, trong hệ thống đa luồng, giữa thời điểm gọi get() và containsKey(), một luồng khác có thể đã xóa hoặc sửa đổi key đó, dẫn đến sai lệch trạng thái cuộc đua (race condition).

AdvancedMapUsage.java Java 17/21 
import java.util.Objects;
import java.util.concurrent.ConcurrentHashMap;

public class AdvancedMapUsage {

    public static final class ApiKey {
        private final String clientId;
        private final String secretHash;

        public ApiKey(String clientId, String secretHash) {
            this.clientId = Objects.requireNonNull(clientId);
            this.secretHash = Objects.requireNonNull(secretHash);
        }

        @Override
        public boolean equals(Object o) {
            if (this == o) return true;
            if (o == null || getClass() != o.getClass()) return false;
            ApiKey apiKey = (ApiKey) o;
            return Objects.equals(clientId, apiKey.clientId) && 
                   Objects.equals(secretHash, apiKey.secretHash);
        }

        @Override
        public int hashCode() {
            // Sử dụng nhân tử số nguyên tố 31 để phân phối hash tối ưu nhất
            int result = clientId.hashCode();
            result = 31 * result + secretHash.hashCode();
            return result;
        }
    }

    private final ConcurrentHashMap<ApiKey, String> tokenCache = new ConcurrentHashMap<>();

    public String getOrGenerateToken(ApiKey key) {
        // Dùng computeIfAbsent để chạy Thread-Safe, chỉ băm và tính toán nếu key chưa tồn tại
        return tokenCache.computeIfAbsent(key, k -> {
            System.out.println("Generating secure token for Client: " + k.clientId);
            return Long.toHexString(Double.doubleToLongBits(Math.random()));
        });
    }
}

2. Java Concurrency & Multi-threading trên Production

Concurrency

2.1. Vận hành ThreadPoolExecutor trên Production

Một lỗi cực kỳ phổ biến của Java Developer là khởi tạo Thread Pool bằng các method mặc định như Executors.newCachedThreadPool() hay Executors.newFixedThreadPool(). Cơ chế hoạt động của ThreadPoolExecutor:

Khi có task mới đến:
1. Nếu số luồng đang chạy nhỏ hơn corePoolSize -> Tạo luồng mới lập tức.
2. Nếu đạt đến corePoolSize -> Đẩy task vào hàng đợi BlockingQueue.
3. Nếu hàng đợi bị đầy và số luồng hiện tại nhỏ hơn maximumPoolSize -> Tạo luồng tạm thời mới (được giải phóng sau thời gian keepAliveTime).
4. Nếu hàng đợi đầy và số luồng đã đạt tối đa -> Kích hoạt RejectedExecutionHandler.

Cảnh báo OOM: Executors.newFixedThreadPool() sử dụng một LinkedBlockingQueue không giới hạn kích thước (Unbounded - mặc định dung lượng là Integer.MAX_VALUE). Khi luồng làm việc không xử lý kịp hoặc DB bị nghẽn, các task chờ xếp hàng liên tục trong hàng đợi cho đến khi bộ nhớ Heap cạn kiệt, dẫn đến lỗi java.lang.OutOfMemoryError: Java heap space.

2.2. Lập trình bất đồng bộ không nghẽn với CompletableFuture

Khi cần tổng hợp dữ liệu từ nhiều nguồn API độc lập (ví dụ: lấy thông tin User, ví điện tử, và lịch sử đơn hàng), việc gọi tuần tự (blocking) sẽ làm tăng đáng kể độ trễ (latency). Ta kết hợp nhiều task không nghẽn như sau:

AsyncPipelineDemo.java Java 17 
import java.util.concurrent.*;

public class AsyncPipelineDemo {

    private final ExecutorService customPool = new ThreadPoolExecutor(
            10, 50, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000), // Giới hạn queue tránh OOM
            new ThreadPoolExecutor.CallerRunsPolicy() // Quá tải tự chạy trên Thread gọi
    );

    public record UserProfile(String name, double balance, String lastOrder) {}

    public UserProfile fetchFullProfile(String userId) {
        // 1. Khởi chạy song song 3 tác vụ độc lập
        CompletableFuture<String> nameFuture = CompletableFuture.supplyAsync(
                () -> callUserService(userId), customPool);
        
        CompletableFuture<Double> balanceFuture = CompletableFuture.supplyAsync(
                () -> callWalletService(userId), customPool);
                
        CompletableFuture<String> orderFuture = CompletableFuture.supplyAsync(
                () -> callOrderService(userId), customPool);

        // 2. Gộp kết quả không chặn (Non-blocking Join)
        CompletableFuture<Void> combinedFuture = CompletableFuture.allOf(
                nameFuture, balanceFuture, orderFuture);

        // Đợi cả 3 luồng hoàn tất
        combinedFuture.join(); 

        try {
            return new UserProfile(nameFuture.get(), balanceFuture.get(), orderFuture.get());
        } catch (InterruptedException | ExecutionException e) {
            throw new RuntimeException("Failed to aggregate profile data", e);
        }
    }

    private String callUserService(String id) { return "John Doe"; }
    private double callWalletService(String id) { return 1500.50; }
    private String callOrderService(String id) { return "MacBook Pro M3"; }
}

2.3. Project Loom - Virtual Threads (Java 21 LTS)

Trước Java 21, mô hình xử lý là Thread-per-request. Mỗi Thread trong Java là một Platform Thread, liên kết trực tiếp tỉ lệ 1:1 với một luồng hệ điều hành vật lý (OS Thread). Điều này gây ra hạn chế cực lớn:

  • Mỗi Platform Thread tiêu tốn khoảng 1MB bộ nhớ ngoài heap cho call stack. Hệ thống không thể scale lên vượt quá vài ngàn luồng đồng thời.
  • Khi thực hiện các tác vụ nghẽn I/O (I/O Bound) như truy vấn SQL hoặc gọi API từ xa, luồng OS vật lý bị đưa vào trạng thái ngủ (Waiting), gây lãng phí tài nguyên CPU nghiêm trọng.

Kiến trúc Virtual Threads (Luồng ảo) hoạt động ra sao?

Virtual Thread là một thực thể nhẹ chạy trong vùng nhớ Heap của JVM chứ không phải luồng OS. JVM quản lý việc chạy hàng triệu luồng ảo trên một số ít các Platform Threads (gọi là Carrier Threads - thường bằng số nhân CPU vật lý).

  • Cơ chế Continuation: Khi một Virtual Thread chạy đến lệnh I/O block (ví dụ: gửi một request HTTP), JVM sẽ tự động ngắt thực thi luồng ảo đó, chụp lại toàn bộ ngăn xếp trạng thái (Execution Frame State) và lưu trữ nó vào Heap (gọi là cơ chế Yield).
  • Carrier Thread lúc này được giải phóng ngay lập tức để thực thi các tác vụ khác.
  • Khi dữ liệu I/O phản hồi thành công, JVM sẽ lên lịch đưa trạng thái cũ từ Heap quay trở lại một Carrier Thread trống để tiếp tục chạy (gọi là cơ chế Resume).
VirtualThreadsDemo.java Java 21 
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.util.concurrent.Executors;
import java.util.stream.IntStream;

public class VirtualThreadsDemo {

    public void runConcurrently() {
        // Sử dụng ExecutorService tự tạo một luồng ảo trên mỗi Task (Không cần Thread Pool để tái sử dụng luồng)
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            HttpClient client = HttpClient.newHttpClient();
            
            // Kích hoạt 10,000 requests song song không làm nghẽn luồng OS
            IntStream.range(0, 10000).forEach(i -> {
                executor.submit(() -> {
                    HttpRequest request = HttpRequest.newBuilder()
                            .uri(URI.create("https://httpbin.org/delay/1"))
                            .build();
                    // Khối lệnh blocking I/O dưới đây sẽ tự động kích hoạt Yield giải phóng Carrier Thread
                    HttpResponse<String> response = client.send(
                            request, HttpResponse.BodyHandlers.ofString());
                    System.out.println("Task " + i + " status: " + response.statusCode());
                    return response.body();
                });
            });
        } // Tự động đóng executor và đợi toàn bộ tác vụ ảo hoàn tất
    }
}

3.2. String.intern() & Bẫy hiệu năng với StringTableSize

Khi gọi s.intern(), JVM kiểm tra xem chuỗi có nội dung tương đương đã tồn tại trong String Pool chưa. Nếu có, nó trả về tham chiếu của đối tượng trong pool. Nếu chưa, đối tượng String đó được đưa vào pool.

  • Cơ chế hoạt động: String Pool bản chất là một cấu trúc bảng băm (String Table) dạng mở (Open Addressing Hash Map) có kích thước cố định được cấu hình thông qua tham số khởi động: -XX:StringTableSize.
  • Bẫy hiệu năng: Nếu hệ thống thực hiện `intern()` hàng triệu chuỗi khác nhau mà không nâng kích thước bảng này (mặc định trong Java 8-17 khoảng 60,013 buckets), số lượng va chạm mã băm sẽ cực kỳ cao. Lúc này mỗi bucket sẽ chứa các danh sách liên kết cực kỳ dài, biến thao tác tìm kiếm từ O(1) thành tuần tự O(N). Điều này làm gia tăng mạnh thời gian dừng ứng dụng để dọn rác (GC pauses) và làm chậm toàn bộ hệ thống.
  • Giải pháp Senior: Nếu bắt buộc phải lưu cache dữ liệu dạng chuỗi lớn, hãy nâng kích thước bằng tham số: -XX:StringTableSize=1000003 (sử dụng một số nguyên tố lớn để giảm va chạm).

3.3. Quy trình điều tra rò rỉ bộ nhớ (Memory Leak)

Rò rỉ bộ nhớ trong Java xảy ra khi các đối tượng không còn được sử dụng trong nghiệp vụ nhưng vẫn được giữ tham chiếu bởi các đối tượng sống lâu dài khác (như biến static, luồng chạy nền), khiến bộ thu gom rác (GC) không thể dọn dẹp.

Các nguyên nhân gây Memory Leak phổ biến:

  1. Lạm dụng biến Static: Biến static tồn tại cùng tuổi thọ với ClassLoader. Một static Map lưu cache dữ liệu nếu không có cơ chế dọn dẹp (TTL, LRU) sẽ phình to liên tục.
  2. ThreadLocal không gọi remove(): Trong Spring Boot/Tomcat, các luồng (Thread) trong Thread Pool được tái sử dụng liên tục cho các request khác nhau. Nếu ghi dữ liệu vào ThreadLocal mà không gọi remove() trong khối finally, dữ liệu của request cũ sẽ bị giữ mãi trên luồng đó.
  3. Không đóng tài nguyên (Unclosed resources): Các kết nối Database, HTTP client streams, file handles không được bao bọc trong khối try-with-resources.

Quy trình điều tra thực tế sử dụng Eclipse MAT (Memory Analyzer Tool):

  1. Xuất Heap Dump khi hệ thống sắp sập (OOM): Cấu hình JVM tham số: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/logs/app.hprof. Hoặc chủ động chụp thủ công bằng lệnh: jcmd <pid> GC.heap_dump /tmp/heap.hprof.
  2. Nạp file Dump vào Eclipse MAT: Chạy phân tích "Leak Suspects Report".
  3. Kiểm tra Histogram và Dominator Tree: Xác định Class nào đang chiếm dụng số lượng byte lớn nhất trong bộ nhớ Heap.
  4. Tra cứu đường dẫn GC Roots (Exclude Weak/Soft References): Nhấp chuột phải vào class nghi ngờ -> Path To GC Roots -> exclude all phantom/weak/soft references. MAT sẽ hiển thị sơ đồ đường dẫn tham chiếu trực tiếp đến đối tượng sống nào đang kìm giữ không cho GC dọn dẹp. Từ đó phát hiện ra dòng code lỗi.
ThreadLocalLeakPrevention.java Java 17 / Spring Best Practice 
import javax.servlet.*;
import java.io.IOException;

public class ThreadLocalLeakPrevention implements Filter {

    public static final class RequestContext {
        private final String transactionId;

        public RequestContext(String transactionId) {
            this.transactionId = transactionId;
        }
    }

    // Biến ThreadLocal lưu trữ thông tin request trên từng luồng
    private static final ThreadLocal<RequestContext> CONTEXT_HOLDER = new ThreadLocal<>();

    public static void setContext(RequestContext context) {
        CONTEXT_HOLDER.set(context);
    }

    public static RequestContext getContext() {
        return CONTEXT_HOLDER.get();
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        try {
            String txnId = request.getParameter("transaction_id");
            setContext(new RequestContext(txnId));
            
            // Thực thi luồng xử lý tiếp theo
            chain.doFilter(request, response);
        } finally {
            // [BẮT BUỘC] Giải phóng ThreadLocal ở khối finally để tránh rò rỉ khi luồng quay về Thread Pool
            CONTEXT_HOLDER.remove();
        }
    }
}

4. JVM Garbage Collection (GC) & Tuning hiệu năng

Garbage Collection

4.1. Nguyên lý phân chia vùng nhớ thế hệ (Generational Hypothesis)

JVM chia bộ nhớ Heap thành hai phân vùng chính dựa trên giả định thực tế: hầu hết các đối tượng chỉ tồn tại trong thời gian ngắn ngủi rồi chết đi nhanh chóng.

  • Young Generation: Chứa các đối tượng mới sinh. Chia thành 3 vùng nhỏ: **Eden**, **Survivor S0**, và **Survivor S1** (tỉ lệ mặc định 8:1:1).
    • Đối tượng mới khởi tạo nằm ở Eden. Khi Eden đầy, Minor GC diễn ra. Các đối tượng còn sống được chuyển sang S0 (hoặc S1), độ tuổi (Tenuring Threshold) tăng lên 1.
    • Quá trình lặp lại, luân phiên dọn dẹp giữa S0 và S1. Mỗi lần sống sót qua Minor GC, tuổi của đối tượng tăng lên 1.
  • Old Generation (Tenuring Promotion): Khi độ tuổi đạt ngưỡng tối đa (cấu hình qua -XX:MaxTenuringThreshold, mặc định là 15), đối tượng được chuyển hẳn (promote) sang vùng Old Generation để lưu giữ lâu dài. Khi Old Generation đầy, **Major GC** (hoặc Full GC) sẽ được kích hoạt để dọn dẹp.

4.2. So sánh các thuật toán GC thế hệ mới trên Production

Hệ thống chịu tải thực tế cần cấu hình GC tối ưu hóa giữa băng thông (Throughput) và độ trễ dừng thế giới (STW - Stop-The-World Latency):

Thuật toán GC Cơ chế hoạt động Ưu điểm Nhược điểm & Ngữ cảnh sử dụng
G1GC (Garbage-First) Chia bộ nhớ Heap thành hàng ngàn phân vùng nhỏ độc lập (Regions). Nó thống kê phân vùng nào chứa nhiều rác nhất để gom trước nhằm đạt mục tiêu giới hạn thời gian trễ do người dùng cài đặt. Hạn chế thời gian Stop-The-World tối đa theo cấu hình. Phù hợp cho Heap tầm trung (từ 4GB đến 64GB). Độ trễ STW vẫn rơi vào khoảng từ 50ms đến 200ms tùy kích thước heap. Cấu hình khởi động: -XX:+UseG1GC -XX:MaxGCPauseMillis=200.
ZGC (Zero Garbage Collector - Java 21) Sử dụng cơ chế Colored Pointers (con trỏ màu) kết hợp Load Barriers (rào cản tải). Nó thực hiện việc đánh dấu và di dời đối tượng dọn dẹp song song (concurrent) với các luồng ứng dụng mà không cần dừng hệ thống. Độ trễ dừng hệ thống cực kỳ nhỏ (luôn nhỏ hơn 1ms), không phụ thuộc vào kích thước của Heap. Hỗ trợ Heap từ 16GB lên tới hàng Terabyte. Tiêu thụ thêm năng lượng CPU của hệ thống để chạy dọn rác nền song song. Khuyên dùng từ Java 21 cho các API yêu cầu phản hồi siêu tốc. Kích hoạt: -XX:+UseZGC.

4.3. Cách đọc log GC & Tuning cơ bản

Khi bật log GC bằng tham số -Xlog:gc*,gc+phases=debug:file=/var/log/gc.log:time,uptime,pid:filecount=5,filesize=100M, ta cần quan sát:

  • GC (Allocation Failure): Cho biết GC bị kích hoạt vì cạn kiệt bộ nhớ không thể cấp phát thêm đối tượng mới.
  • Pause Young (Normal) (G1 Evacuation Pause): Thời gian dừng để dọn vùng Young Gen. Nếu vượt quá mong đợi, cần tối ưu hóa lại tham số.
  • [GC concurrent-mark]: Thời gian đánh dấu rác chạy ngầm song song với code của bạn (không block app).

5. Lớp Tiện ích (Wrapper Classes), Đa hình Runtime & Cơ chế Reflection API

Java Core Internals

5.1. Cơ chế Autoboxing/Unboxing & Bẫy Integer Cache

Autoboxing là cơ chế tự động chuyển đổi từ kiểu dữ liệu nguyên thủy (Primitive) sang đối tượng Wrapper tương ứng (ví dụ: int thành Integer). Quá trình này được JVM thực thi bằng cách gọi ngầm phương thức Integer.valueOf(int).

  • Integer Cache: Để tối ưu hóa tài nguyên, JVM duy trì một bộ nhớ đệm chứa các đối tượng Integer có giá trị từ -128 đến 127. Khi ta gọi Integer.valueOf(x) với giá trị nằm trong khoảng này, JVM sẽ trả về tham chiếu đến đối tượng có sẵn trong bộ nhớ đệm mà không tạo đối tượng mới.
  • Tác hại hiệu năng: Khi xử lý dữ liệu lớn (ví dụ: vòng lặp duyệt qua hàng triệu bản ghi), việc tính toán trực tiếp trên các biến Wrapper sẽ ép JVM thực hiện Autoboxing liên tục, tạo ra hàng triệu đối tượng rác ngắn hạn trên Heap, làm tăng áp lực dọn rác lên GC và tiêu tốn bộ nhớ gấp 4 lần so với kiểu nguyên thủy.

5.2. Đa hình lúc Runtime: Virtual Method Table (vtable) & Dynamic Binding

Khi ta gọi một phương thức bị ghi đè (Overridden Method) thông qua một tham chiếu lớp cha, làm thế nào JVM xác định chính xác phiên bản phương thức của lớp con để chạy tại runtime? Cơ chế thực thi:

  1. Khi một lớp (Class) được tải vào bộ nhớ Metaspace, JVM sẽ xây dựng một Bảng Phương thức Ảo (vtable - Virtual Method Table) cho lớp đó.
  2. Bảng vtable chứa danh sách các con trỏ trỏ trực tiếp đến mã máy thực thi của từng phương thức mà lớp đó sở hữu hoặc kế thừa từ lớp cha.
  3. Nếu lớp con ghi đè một phương thức của lớp cha, địa chỉ con trỏ tương ứng trong vtable của lớp con sẽ được cập nhật trỏ thẳng đến mã của lớp con thay vì mã lớp cha.
  4. Tại thời điểm runtime (Dynamic Binding), khi luồng chạy đến lệnh gọi hàm, JVM sẽ dựa vào kiểu đối tượng thực tế trên Heap (chứ không phải kiểu khai báo của biến ở Stack) để tra cứu trong bảng vtable và nhảy thẳng đến mã thực thi chính xác. Quá trình tra cứu vtable có độ phức tạp là O(1).

5.3. Reflection API & Cơ chế Spring @Transactional Proxy

Reflection API cho phép kiểm tra cấu trúc class, đọc annotation, truy cập các biến private lúc runtime. Tuy nhiên, thao tác này có tốc độ chậm hơn gọi code trực tiếp vì JVM không thể áp dụng các cơ chế tối ưu hóa mã hóa (JIT Compiler optimization).

Cơ chế hoạt động của @Transactional trong Spring: Spring Framework sử dụng Proxy Pattern kết hợp Reflection để quản lý giao dịch:

Client Code ---> [ Spring Generated Proxy Object ] ---> Target Service class (@Transactional)
                        |                                       |
                        |-- 1. Mở Transaction                   |
                        |-- 2. Gọi Method của Target ---------> | (Thực thi Business Logic)
                        |                                       |
                        |-- 3. Commit (Nếu thành công)          |
                        |-- 4. Rollback (Nếu xảy ra RuntimeException)
  • Khi một Bean được đánh dấu @Transactional, Spring sẽ bọc Bean đó lại bên trong một đối tượng Proxy động (sử dụng JDK Dynamic Proxy hoặc CGLIB).
  • Khi Client gọi method, nó thực chất đang gọi method của đối tượng Proxy.
  • Proxy mở kết nối CSDL và bắt đầu Transaction. Tiếp theo, nó sử dụng Reflection để gọi phương thức nghiệp vụ thực tế của Target Bean.
  • Nếu phương thức chạy thành công, Proxy thực hiện commit(). Nếu ném ra một RuntimeException, Proxy tự động gọi rollback() để bảo toàn dữ liệu.
ReflectionProxySimulation.java Java Custom Transaction Simulator 
import java.lang.annotation.*;
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

public class ReflectionProxySimulation {

    @Retention(RetentionPolicy.RUNTIME)
    @Target(ElementType.METHOD)
    public @interface CustomTransaction {}

    public interface PaymentService {
        void processPayment(double amount);
    }

    public static class PaymentServiceImpl implements PaymentService {
        @Override
        @CustomTransaction
        public void processPayment(double amount) {
            System.out.println("Executing database write for amount: " + amount);
        }
    }

    public static class TransactionInvocationHandler implements InvocationHandler {
        private final Object target;

        public TransactionInvocationHandler(Object target) {
            this.target = target;
        }

        @Override
        public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
            // Tìm kiếm xem method ở lớp thực thi thực tế có chứa annotation hay không
            Method targetMethod = target.getClass().getMethod(method.getName(), method.getParameterTypes());
            
            if (targetMethod.isAnnotationPresent(CustomTransaction.class)) {
                System.out.println("[Proxy] >>> START Transaction");
                try {
                    Object result = method.invoke(target, args); // Sử dụng Reflection để chạy method thực tế
                    System.out.println("[Proxy] >>> COMMIT Transaction");
                    return result;
                } catch (Exception e) {
                    System.out.println("[Proxy] >>> ROLLBACK Transaction due to: " + e.getCause());
                    throw e.getCause();
                }
            }
            return method.invoke(target, args);
        }
    }

    public static void main(String[] args) {
        PaymentService target = new PaymentServiceImpl();
        PaymentService proxyInstance = (PaymentService) Proxy.newProxyInstance(
                PaymentService.class.getClassLoader(),
                new Class<?>[]{PaymentService.class},
                new TransactionInvocationHandler(target)
        );

        proxyInstance.processPayment(99.99);
    }
}

6. Bộ câu hỏi phỏng vấn Q&A thực chiến cấp độ Senior / Architect

Global Prep Q&A
Thiết kế một giải pháp lưu bộ nhớ đệm (Caching System) có tốc độ phản hồi cực cao, chịu tải hàng triệu lượt ghi đồng thời trên một node đơn lẻ. Bạn sẽ lựa chọn ConcurrentHashMap, Redis cục bộ hay thư viện Caffeine Cache? Hãy phân tích các khía cạnh về mặt bộ nhớ và chi phí CPU.
Đây là bài toán so sánh giữa cache trực tiếp trong Heap Java với cache ngoài tiến trình (out-of-process) và cache có cơ chế dọn dẹp:
  • Redis cục bộ: Đảm bảo dữ liệu không bị mất khi ứng dụng Java khởi động lại, nhưng gặp điểm nghẽn về hiệu năng do chi phí truyền tải qua mạng (Network I/O) và tuần tự hóa dữ liệu (Serialization/Deserialization) giữa Java App và Redis. Không tối ưu nhất cho hiệu năng tính bằng microsecond trên một node đơn lẻ.
  • ConcurrentHashMap: Tốc độ truy xuất nhanh nhất vì dữ liệu nằm trực tiếp trong bộ nhớ Heap của JVM dưới dạng Object gốc. Tuy nhiên, nó không hỗ trợ cơ chế tự động dọn dẹp khi đầy (Eviction Policy như LRU, LFU) và không hỗ trợ thời hạn hết hạn (TTL). Nếu dữ liệu ghi vào liên tục không kiểm soát, nó sẽ phình to gây lỗi tràn bộ nhớ (OOM).
  • Caffeine Cache: Đây là lựa chọn tối ưu nhất cho bài toán này. Caffeine là một thư viện In-memory cache hiệu năng cực cao viết bằng Java. Nó sử dụng cấu trúc dữ liệu băm kết hợp với thuật toán **Window TinyLFU** (thuật toán lọc tần suất xuất hiện tối ưu hơn LRU thông thường) để duy trì kích thước cache cố định và loại bỏ phần tử cực kỳ thông minh. caffeine sử dụng các bộ đệm ghi vòng (Ring Buffers) không khóa để tối thiểu hóa tranh chấp luồng ghi, cho phép scale tuyến tính theo số nhân CPU.
Trong Java 21, khi sử dụng Virtual Threads để tối ưu hóa tác vụ I/O, điều gì xảy ra nếu đoạn code nghiệp vụ của bạn chứa khối lệnh "synchronized" và gọi truy vấn xuống Cơ sở dữ liệu? Bạn sẽ xử lý lỗi này thế nào để hệ thống không bị nghẽn?
Khi một luồng ảo (Virtual Thread) thực thi bên trong một khối hoặc phương thức có từ khóa synchronized, hoặc khi nó gọi một hàm native bên ngoài JVM, hiện tượng Thread Pinning (Ghim luồng) sẽ xảy ra.
  • Hậu quả: Khi luồng ảo bị ghim, JVM không thể nhấc trạng thái của nó ra để cất vào Heap khi gặp lệnh chặn I/O nữa. Lúc này, luồng OS vật lý bên dưới (Carrier Thread) sẽ bị khóa cứng hoàn toàn trong trạng thái ngủ. Nếu tất cả các Carrier Threads của JVM đều bị ghim do đụng độ synchronized, hệ thống sẽ rơi vào trạng thái nghẽn cổ chai hoàn toàn, làm mất sạch lợi thế về hiệu năng của Virtual Threads.
  • Giải pháp khắc phục: Ta phải thay thế toàn bộ các khối khóa tĩnh synchronized bằng ReentrantLock của gói thư viện java.util.concurrent.locks. Khóa ReentrantLock được viết bằng mã Java thuần túy và được JVM tối ưu hóa để tương tác mượt mà với Virtual Threads. Khi luồng ảo gặp khóa ReentrantLock đang bị giữ, nó vẫn nhường (yield) Carrier Thread cho luồng khác chạy bình thường mà không gây ra hiện tượng ghim luồng.
Hãy đề xuất cấu hình JVM cụ thể cho hai hệ thống sau: (1) Hệ thống Pipeline xử lý Batch đồng bộ dữ liệu khối lượng cực lớn, ưu tiên lưu lượng xử lý tối đa (Throughput); và (2) Hệ thống REST API phục vụ thanh toán tiền tệ, yêu cầu thời gian phản hồi siêu tốc, độ trễ cực thấp (Low Latency).
Đây là bài toán điều chỉnh (tuning) JVM tùy thuộc vào mục tiêu nghiệp vụ:
  1. Hệ thống xử lý Batch đồng bộ dữ liệu (Throughput-oriented):
    • Thuật toán GC: Sử dụng **Parallel GC** (kích hoạt bằng -XX:+UseParallelGC). Thuật toán này sử dụng toàn bộ CPU để thực hiện việc thu gom rác một cách song song nhanh nhất có thể. Dù thời gian dừng STW dài hơn, nhưng tổng thời gian CPU dành cho tính toán nghiệp vụ (Throughput) là lớn nhất.
    • Cấu hình bộ nhớ: Đặt kích thước Heap lớn và cố định ngay từ đầu để tránh chi phí co giãn Heap: -Xms16g -Xmx16g. Cài đặt tỉ lệ vùng New Gen lớn để chứa các đối tượng trung gian: -XX:NewRatio=2 (Young Gen chiếm 1/3 tổng Heap).
  2. Hệ thống REST API thanh toán (Latency-oriented):
    • Thuật toán GC: Sử dụng **ZGC** (kích hoạt bằng -XX:+UseZGC, và bật tính năng Generational ZGC của Java 21 bằng -XX:+ZGenerational). ZGC đảm bảo thời gian tạm dừng của luồng ứng dụng nhỏ hơn 1ms, loại bỏ hoàn toàn hiện tượng nghẽn mạng do GC tạm dừng ứng dụng.
    • Cấu hình bộ nhớ: Đặt -Xms12g -Xmx12g. Cấu hình kích thước tạm dừng mục tiêu cực thấp: -XX:MaxGCPauseMillis=10. Khóa toàn bộ bộ nhớ Heap vật lý để hệ điều hành không swap bộ nhớ xuống ổ đĩa ảo bằng tham số: -XX:+AlwaysPreTouch.
Một hệ thống xử lý đa khách hàng (Multi-tenant) lưu giữ hàng triệu thông tin cấu hình của các tenant trong một ConcurrentHashMap toàn cục. Qua phân tích Heap Dump, bạn nhận thấy dung lượng bộ nhớ bị phình to bất thường, và GC liên tục chạy Full GC làm chậm hệ thống. Hãy chỉ ra các lỗi thiết kế có thể có và cách tái cấu trúc hệ thống này.
Lưu trữ dữ liệu phi cấu trúc vô hạn trong một map tĩnh toàn cục là một thiết kế phản mẫu (anti-pattern) kinh điển dẫn đến Memory Leak và cạn kiệt tài nguyên. Các lỗi thiết kế cụ thể:
  • Không có cơ chế dọn dẹp (Eviction Policy): Bản chất ConcurrentHashMap giữ tham chiếu mạnh (Strong References) đến các key và value. Nếu một khách hàng (tenant) không còn hoạt động, dữ liệu cấu hình của họ vẫn nằm mãi trong map và không bao giờ được giải phóng.
  • Cách tái cấu trúc hệ thống:
    1. Thay thế ConcurrentHashMap bằng một thư viện cache chuyên dụng hỗ trợ giới hạn dung lượng và thời gian sống của phần tử, ví dụ **Caffeine Cache** cấu hình thời gian hết hạn hợp lý: expireAfterAccess(30, TimeUnit.MINUTES) hoặc maximumSize(5000).
    2. Nếu vẫn bắt buộc dùng cấu trúc thô sơ, ta có thể bao bọc các đối tượng cấu hình bằng **WeakReference** hoặc **SoftReference** thông qua lớp WeakHashMap. Tuy nhiên, ConcurrentHashMap không hỗ trợ Weak Keys trực tiếp, nên ta phải tự tạo một Thread-Safe Weak Wrapper hoặc sử dụng Cache ngoài tiến trình như **Redis** để lưu trữ cấu hình nếu dung lượng quá lớn vượt quá giới hạn RAM của một JVM đơn lẻ.
Khi kiểm thử chịu tải (Load Test) một dịch vụ Spring Boot xử lý JSON Payload lớn, bạn quan sát thấy CPU của server bị spike lên 100%, thời gian phản hồi tăng vọt từ 20ms lên 5000ms. Sử dụng công cụ giám sát, bạn thấy tiến trình Java liên tục chạy Minor GC và thỉnh thoảng có Major GC. Hãy giải thích nguyên nhân sâu xa ở cấp độ quản lý bộ nhớ JVM và cách tối ưu hóa ở cả code và JVM.
Đây là hiện tượng **GC Thrashing** (Quá tải dọn rác).
  • Nguyên nhân sâu xa: Khi xử lý các JSON Payload lớn, thư viện ObjectMapper (Jackson) hoặc mã ứng dụng liên tục phân tích và sinh ra hàng loạt các đối tượng String, Map, List trung gian ngắn hạn. Do tốc độ request gửi đến cực lớn, tốc độ sinh đối tượng rác ở vùng **Eden** (Young Gen) vượt xa tốc độ dọn dẹp của Minor GC. Eden nhanh chóng bị lấp đầy liên tục.
  • Điều này ép JVM phải chạy Minor GC dồn dập. Vì Minor GC chạy liên tục, các đối tượng chưa kịp chết hoàn toàn đã đạt độ tuổi khuyến mãi và bị đẩy (promote) cưỡng bức lên **Old Gen** sớm hơn thời hạn (hiện tượng *Premature Promotion*). Khi vùng Old Gen bị lấp đầy bởi các đối tượng rác chưa kịp dọn này, JVM bắt buộc phải kích hoạt tiến trình nặng nề nhất là **Full GC (Stop-The-World)** làm đóng băng toàn bộ các luồng xử lý nghiệp vụ, gây nghẽn cổ chai và tăng vọt độ trễ phản hồi của API.
  • Giải pháp khắc phục:
    1. Code-level: Tận dụng cơ chế đọc luồng (Streaming API của Jackson) thay vì đọc toàn bộ JSON Payload vào bộ nhớ cùng lúc. Tái sử dụng các cấu trúc mảng đệm bằng cách sử dụng các Object Pools nếu cần thiết. Tránh thực hiện các phép cộng chuỗi lớn trong vòng lặp, thay thế bằng StringBuilder.
    2. JVM Tuning: Tăng kích thước vùng bộ nhớ New Gen để đảm bảo các đối tượng ngắn hạn có đủ thời gian và không gian để chết hẳn ở vùng Young Gen trước khi bị đẩy lên Old Gen: sử dụng tham số -XX:NewRatio=1 hoặc tăng tổng Heap -Xmx. Chuyển đổi sang sử dụng **ZGC** để giảm thời gian tạm dừng dọn rác xuống dưới 1ms, tránh bị nghẽn tắc luồng xử lý do Full GC gây ra.
