Tuyển dụng & Chiến lược Migration

Bộ Câu Hỏi Phỏng Vấn Senior Java
Migrate Database từ MongoDB sang PostgreSQL

Tài liệu phỏng vấn lập trình viên Senior Java / System Architect tập trung vào chiến lược di chuyển dữ liệu lớn không gián đoạn dịch vụ (Zero-Downtime Migration) cho phân hệ mối quan hệ người dùng user_relation của dự án main-server.

Bối cảnh hệ thống thực tế (Case Study)

Chiến dịch Migration phân hệ User Relation

Hệ thống mạng xã hội Live-streaming đang lưu trữ dữ liệu mối quan hệ giữa người dùng (Follow, Block, Friend) trong MongoDB. Do yêu cầu về ràng buộc dữ liệu toàn vẹn (Foreign Key) và tối ưu hóa các truy vấn kết hợp phức tạp (như Mutual Friends, Connection Degrees), chúng ta quyết định di chuyển dữ liệu này sang PostgreSQL.

Hiện tại, dự án main-server mới chỉ triển khai phần di chuyển cho module user_relation. Thử thách lớn nhất là phải di chuyển và chuyển đổi hệ thống lưu trữ dưới áp lực hàng triệu request đọc/ghi mỗi ngày mà **không được gây gián đoạn dịch vụ (Zero-downtime)** và **không làm mất mát dữ liệu**.

Chiến lược cốt lõi:
Sử dụng mô hình **Dual-Write** (Ghi kép vào cả 2 DB) phối hợp với **Shadow-Read & Compare** (Đọc từ cả 2 DB để so khớp đối chiếu chất lượng dữ liệu ngầm) được cấu hình động thông qua hệ thống **Feature Flags (Dynamic Configuration Flags)**.
Kịch bản Phỏng Vấn Mẫu (STAR Template)

Mẫu trả lời phỏng vấn: "Hãy chia sẻ kinh nghiệm di chuyển Database (Migration) lớn nhất em từng làm?"

Kịch bản mẫu giúp ứng viên/người phỏng vấn cấu trúc câu trả lời chuyên nghiệp dựa trên module user_relation thực tế trong main-server:

1. Dẫn dắt (Khẳng định vai trò)

"Dạ rồi ạ. Tại dự án trước, em đã trực tiếp thiết kế và chỉ đạo chiến dịch di chuyển dữ liệu mối quan hệ người dùng (User Relation Subsystem - gồm follow, block, bạn bè) với hơn 50 triệu bản ghi từ MongoDB sang PostgreSQL mà không có bất kỳ giây downtime nào cho dịch vụ."

2. Tình huống & Nhiệm vụ (Situation / Task)

"Hệ thống ban đầu lưu dữ liệu Follow/Block trên MongoDB. Do thiếu ràng buộc khóa ngoại cứng ở tầng DB, khi hệ thống chạy lâu ngày xuất hiện các dữ liệu rác (ví dụ ID người dùng bị follow không còn tồn tại trong bảng Users). Việc truy vấn tìm kiếm bạn bè chung (Mutual follows) cũng gây tải rất nặng do MongoDB không tối ưu hóa các phép toán Join.

Nhiệm vụ của em là di chuyển toàn bộ dữ liệu này sang PostgreSQL để áp dụng tính toàn vẹn dữ liệu và tối ưu hóa hiệu năng, đồng thời đảm bảo quá trình di chuyển hoàn toàn thông suốt, người dùng vẫn thực hiện follow/block bình thường."

3. Hành động Chi tiết (Action)

"Em đã triển khai giải pháp di chuyển 4 bước cực kỳ chặt chẽ:"

  • Bước 1 - Dual-Write (Ghi kép): "Em sửa code ở module user_relation trong main-server để khi có sự kiện ghi mới (như Follow), ứng dụng sẽ ghi đồng thời vào cả MongoDB và PostgreSQL. Trong đó, MongoDB vẫn là DB chính (Primary) để đảm bảo không lỗi ứng dụng, còn việc ghi PostgreSQL là thứ cấp (Secondary) được bọc trong khối try-catch không gây lỗi lan truyền nếu Postgres gặp sự cố hoặc timeout."
  • Bước 2 - Data Migration (Di chuyển dữ liệu cũ): "Khởi chạy một batch job độc lập để quét toàn bộ dữ liệu lịch sử từ MongoDB sang PostgreSQL. Em sử dụng phân trang Keyset Pagination để tránh kẹt I/O DB, kết hợp chống ghi đè dữ liệu cũ lên dữ liệu mới sinh ra nhờ cơ chế đối chiếu thời gian cập nhật (Timestamp comparison)."
  • Bước 3 - Shadow-Read & Dynamic Comparison (Đọc bóng & Đối chiếu động): "Khi Client gọi API lấy danh sách Follow/Block, em cho hệ thống đọc từ MongoDB và đồng thời trigger một luồng phụ (Shadow Thread) đọc từ PostgreSQL. Hệ thống sử dụng một thuật toán Deep-Compare để so khớp cấu trúc dữ liệu hai bên. Nếu phát hiện lệch, hệ thống log lỗi kèm Metric cụ thể để đội ngũ Dev sửa đổi các điểm dữ liệu không thống nhất. Toàn bộ cơ chế đọc này được quản lý qua các Dynamic Flags (Feature Flags) lưu trong Redis/Config Server để cấu hình chuyển đổi trạng thái đọc/ghi theo thời gian thực mà không cần restart server."
  • Bước 4 - Cutover (Chuyển đổi hoàn toàn): "Sau khi tỷ lệ so khớp Shadow-Read đạt 100% liên tục trong 1 tuần, em bật flag chuyển PostgreSQL làm DB chính, MongoDB làm DB thứ cấp. Sau 3 ngày chạy thử ổn định, em tắt hoàn toàn luồng ghi phụ sang MongoDB."

4. Kết quả (Result)

  • "Di chuyển thành công hơn 50 triệu bản ghi quan hệ người dùng sang PostgreSQL với tỷ lệ sai lệch bằng 0."
  • "Tốc độ truy vấn lấy danh sách bạn bè chung tăng gấp 4 lần nhờ các phép Join tối ưu hóa của PostgreSQL."
  • "Loại bỏ hoàn toàn 100% dữ liệu mồ côi (dữ liệu rác khóa ngoại), đảm bảo tính toàn vẹn nghiệp vụ tuyệt đối."
Phần 1: Giao dịch & Ghi

1. Kiến trúc Dual-Write (Ghi kép an toàn)

Câu hỏi dành cho Ứng viên:
Khi triển khai Dual-Write vào cả MongoDB (Legacy) và PostgreSQL (New), làm thế nào để đảm bảo lỗi phát sinh khi ghi xuống DB mới (PostgreSQL) không được làm gián đoạn luồng nghiệp vụ chính của người dùng đang chạy? Bạn sẽ quản lý Transaction như thế nào?

Đáp án kỳ vọng từ ứng viên Senior:

  • Tách biệt Transaction (Non-blocking Secondary Write): MongoDB đóng vai trò là "Source of Truth" tạm thời. Giao dịch ghi MongoDB phải diễn ra đồng bộ và quyết định trạng thái thành công của API. Việc ghi sang PostgreSQL phải được coi là nhiệm vụ thứ cấp (Secondary Task).
  • Xử lý ngoại lệ độc lập: Bọc luồng ghi PostgreSQL trong khối try-catch riêng biệt. Nếu PostgreSQL bị sập hoặc nghẽn, ứng dụng chỉ ghi nhận log cảnh báo hoặc đẩy id cần đồng bộ bù vào hàng đợi lỗi (DLQ/Redis Queue) để xử lý sau, **không được ném exception ra ngoài** làm rollback giao dịch ghi MongoDB.
  • Xử lý bất đồng bộ: Có thể đẩy sự kiện ghi Postgres thành một tác vụ bất đồng bộ (sử dụng @Async hoặc Spring Event Listener phi đồng bộ) để giảm thiểu tối đa Latency của API Client.

Cách triển khai ghi kép an toàn trong Spring Boot:

RelationDualWriteService.java Java 21 / Spring Boot 
@Service
@Slf4j
public class RelationDualWriteService {
    @Autowired
    private MongoRelationRepository mongoRepo;
    @Autowired
    private PostgresRelationRepository pgRepo;

    @Transactional("mongoTransactionManager")
    public void followUser(String followerId, String followingId) {
        // 1. Ghi chính vào MongoDB (Đồng bộ)
        MongoRelation relation = new MongoRelation(followerId, followingId, new Date());
        mongoRepo.save(relation);

        // 2. Ghi kép vào PostgreSQL (Bọc kỹ tránh làm sập API chính)
        try {
            saveToPostgresAsync(followerId, followingId);
        } catch (Exception e) {
            // Chỉ log cảnh báo, không ném Exception để không rollback Mongo transaction
            log.error("Fail to write PG during Dual-Write for follower: {}, following: {}", followerId, followingId, e);
            // Pro-tip: Đẩy vào Redis Queue / MQ để tiến trình Reconciliation đồng bộ bù sau
            pushToReconcileQueue(followerId, followingId);
        }
    }

    @Async("pgWriteExecutor")
    private void saveToPostgresAsync(String followerId, String followingId) {
        PgRelation pgRelation = new PgRelation(Long.parseLong(followerId), Long.parseLong(followingId));
        pgRepo.save(pgRelation);
    }
}
Phần 2: So khớp dữ liệu

2. Cơ chế Shadow-Read & So sánh dữ liệu phi đồng bộ (Shadow-Read & Deep-Compare)

Câu hỏi dành cho Ứng viên:
Làm thế nào để kiểm chứng dữ liệu được ghi kép vào PostgreSQL hoàn toàn khớp 100% với MongoDB dưới áp lực sử dụng thực tế của người dùng? Hãy thiết kế cơ chế Shadow-Read và so sánh dữ liệu (Data Comparison) trong Java mà không làm tăng thời gian phản hồi (Response Latency) của Client.

Đáp án kỳ vọng từ ứng viên Senior:

  • Thiết kế Shadow-Read: Khi có request đọc dữ liệu (ví dụ: lấy danh sách follow của user), ứng dụng sẽ thực hiện gọi đọc chính từ MongoDB và trả kết quả ngay lập tức cho người dùng.
  • Thực thi bất đồng bộ: Trước khi trả kết quả, hệ thống bắn một tác vụ bất đồng bộ (sử dụng CompletableFuture.runAsync hoặc ThreadPool) để đọc dữ liệu tương ứng từ PostgreSQL.
  • Deep Compare Algorithm: Tiến hành so sánh dữ liệu giữa 2 danh sách kết quả (MongoDB vs PostgreSQL). Do cấu trúc bản ghi hai bên khác nhau (MongoDB dùng String ID, BSON document; PostgreSQL dùng Long ID, Relational entity), ứng viên cần đề xuất mapping về một cấu trúc dữ liệu chung (Data Transfer Object - DTO) và thực hiện so khớp:
    • So sánh số lượng bản ghi (Size).
    • So khớp chéo từng ID quan hệ (ví dụ: sắp xếp danh sách theo ID rồi so sánh, hoặc đưa vào Set để so khớp $O(N)$).
  • Logging & Alerting: Nếu có bất cứ phần tử nào bị lệch hoặc thiếu, ghi nhận log dạng WARN/ERROR kèm chi tiết payload lệch để Dev phân tích.

Cách triển khai Shadow-Read & So sánh trong Java:

RelationShadowReadService.java So sánh phi đồng bộ không block Client 
@Service
@Slf4j
public class RelationShadowReadService {
    @Autowired
    private MongoRelationService mongoService;
    @Autowired
    private PostgresRelationService pgService;
    @Autowired
    private ExecutorService shadowReadExecutorPool;

    public List<RelationDto> getFollowings(String userId) {
        // 1. Đọc chính từ MongoDB (Đảm bảo tốc độ và trải nghiệm người dùng)
        List<RelationDto> mongoResult = mongoService.getFollowings(userId);

        // 2. Bắn tiến trình Shadow-Read đọc Postgres và so khớp phi đồng bộ
        CompletableFuture.runAsync(() -> {
            try {
                List<RelationDto> pgResult = pgService.getFollowingsFromPg(userId);
                compareData(userId, mongoResult, pgResult);
            } catch (Exception e) {
                log.error("Shadow-Read failed for user: {}", userId, e);
            }
        }, shadowReadExecutorPool);

        return mongoResult;
    }

    private void compareData(String userId, List<RelationDto> mongoList, List<RelationDto> pgList) {
        if (mongoList.size() != pgList.size()) {
            log.warn("[MIGRATION-MISMATCH] Size mismatch for user {}. Mongo: {}, PG: {}", userId, mongoList.size(), pgList.size());
            return;
        }

        Set<String> mongoIds = mongoList.stream().map(RelationDto::getFollowingId).collect(Collectors.toSet());
        Set<String> pgIds = pgList.stream().map(RelationDto::getFollowingId).collect(Collectors.toSet());

        if (!mongoIds.equals(pgIds)) {
            log.warn("[MIGRATION-MISMATCH] Content mismatch for user {}. Missing in PG: {}, Missing in Mongo: {}",
                userId, 
                mongoIds.stream().filter(id -> !pgIds.contains(id)).toList(),
                pgIds.stream().filter(id -> !mongoIds.contains(id)).toList()
            );
        }
    }
}
Phần 3: Flag cấu hình

3. Hệ thống Cấu hình chuyển đổi trạng thái động (Dynamic Flags / Feature Flags)

Câu hỏi dành cho Ứng viên:
Trong quá trình di chuyển dữ liệu kéo dài nhiều tuần, làm thế nào để bạn thay đổi trạng thái đọc/ghi (ví dụ: chuyển từ ghi Mongo sang ghi kép, hoặc chuyển từ đọc Mongo sang đọc Postgres) mà không cần build lại code hay khởi động lại hệ thống (Zero-restart)? Bạn thiết kế hệ thống cấu hình động này như thế nào?

Đáp án kỳ vọng từ ứng viên Senior:

  • Sử dụng Feature Flags / Dynamic Configuration: Sử dụng một hệ thống quản lý config tập trung như Redis (lưu cấu hình dạng Hash/KeyValue), Spring Cloud Config Server, ZooKeeper, Consul hoặc một thư viện Feature Flag chuyên nghiệp (như Unleash, LaunchDarkly).
  • Khai báo các Trạng thái (Migration States / State Machine): Định nghĩa các flag cấu hình trạng thái:
    • WRITE_MODE: MONGO_ONLY (Chỉ ghi Mongo) → DUAL_WRITE_MONGO_PRIMARY (Ghi kép, Mongo chính) → DUAL_WRITE_PG_PRIMARY (Ghi kép, Postgres chính) → PG_ONLY (Chỉ ghi Postgres).
    • READ_MODE: MONGO_ONLY → SHADOW_READ_COMPARE (Đọc Mongo, shadow đọc Postgres so khớp) → PG_PRIMARY_SHADOW_MONGO (Đọc Postgres chính, shadow đọc Mongo) → PG_ONLY (Chỉ đọc Postgres).
  • Cơ chế tự động Refresh Config: Ứng dụng Java lắng nghe sự kiện thay đổi từ Redis Pub/Sub hoặc sử dụng cơ chế Polling (đọc config từ Redis cache định kỳ sau mỗi 5-10 giây) để cập nhật trạng thái biến cấu hình trong bộ nhớ RAM mà không cần restart server.

Đoạn mã điều phối luồng đọc/ghi động dựa theo Flag:

MigrationFlagCoordinator.java Chuyển mạch động bằng Feature Flags 
@Component
public class MigrationFlagCoordinator {
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String MIGRATION_STATE_KEY = "config:migration:user-relation";

    public WriteMode getWriteMode() {
        String val = redisTemplate.opsForValue().get(MIGRATION_STATE_KEY + ":write-mode");
        return val != null ? WriteMode.valueOf(val) : WriteMode.MONGO_ONLY;
    }

    public ReadMode getReadMode() {
        String val = redisTemplate.opsForValue().get(MIGRATION_STATE_KEY + ":read-mode");
        return val != null ? ReadMode.valueOf(val) : ReadMode.MONGO_ONLY;
    }
}

// Sử dụng trong Service:
@Autowired
private MigrationFlagCoordinator flagCoordinator;

public void createRelation(String followerId, String followingId) {
    WriteMode mode = flagCoordinator.getWriteMode();
    switch (mode) {
        case MONGO_ONLY -> mongoRepo.save(...);
        case DUAL_WRITE_MONGO_PRIMARY -> {
            mongoRepo.save(...);
            saveToPgAsync(followerId, followingId); // Ghi Postgres ngầm
        }
        case DUAL_WRITE_PG_PRIMARY -> {
            pgRepo.save(...);
            saveToMongoAsync(followerId, followingId); // Ghi Mongo ngầm
        }
        case PG_ONLY -> pgRepo.save(...);
    }
}
Phần 4: Xử lý dữ liệu lớn

4. Di chuyển dữ liệu cũ quy mô lớn (Initial Batch Migration)

Câu hỏi dành cho Ứng viên:
Làm thế nào để di chuyển (Migrate) hàng chục triệu bản ghi mối quan hệ cũ từ MongoDB sang PostgreSQL mà không làm cạn kiệt bộ nhớ JVM và không gây ảnh hưởng tới DB giao dịch đang chạy? Bạn giải quyết thế nào nếu gặp bản ghi trùng lặp (Duplicate Key) do dữ liệu ghi kép sinh ra trước đó?

Đáp án kỳ vọng từ ứng viên Senior:

  • Sử dụng Spring Batch hoặc Custom Pagination: Sử dụng cơ chế Chunk-oriented Processing của Spring Batch hoặc viết mã phân trang Keyset Pagination (quét Mongo sử dụng _id > lastId). Tuyệt đối không load toàn bộ dữ liệu vào JVM RAM (gây OutOfMemoryError).
  • Xử lý xung đột Idempotent (Upsert): Trong quá trình di chuyển dữ liệu cũ, rất có thể bản ghi đó đã được cập nhật hoặc tạo mới thông qua luồng ghi kép (Dual-write) trước đó. Để tránh ném ra lỗi trùng lặp khóa (Unique Key Violation / Duplicate Key Exception) làm sập Batch Job, cần sử dụng câu lệnh ghi đè dạng **Upsert** ở Postgres: 
    INSERT INTO user_relations (follower_id, following_id, created_at)
VALUES (?, ?, ?)
ON CONFLICT (follower_id, following_id) 
DO UPDATE SET created_at = EXCLUDED.created_at;
  • Rate Limiting / Throttling: Cấu hình Thread Pool giới hạn tốc độ đọc/ghi (ví dụ giới hạn ở 2000 records/giây) và chỉ chạy Batch Job vào các khung giờ thấp điểm (từ 1h đến 5h sáng) để bảo vệ hiệu năng máy chủ.
Phần 5: Kiểm tra toàn vẹn

5. Dịch vụ đối soát dữ liệu tự động (Reconciliation Service)

Câu hỏi dành cho Ứng viên:
Sau khi đã bật chế độ Dual-Write và di chuyển dữ liệu cũ xong, làm thế nào bạn chắc chắn dữ liệu giữa hai DB khớp nhau hoàn toàn trước khi ngắt bỏ MongoDB? Thiết kế một cơ chế đối soát dữ liệu (Reconciliation Job) liên tục trong Java để sửa chữa các bản ghi lệch.

Đáp án kỳ vọng từ ứng viên Senior:

  • Thiết kế Reconciliation Service chạy ngầm: Khởi chạy một tiến trình background scheduler (sử dụng Spring @Scheduled hoặc Quartz Job) chạy định kỳ để so quét và đối chiếu dữ liệu giữa MongoDB và PostgreSQL.
  • Giải pháp quét đối chiếu theo khoảng thời gian (Time-window Scan): Mỗi lượt quét, job sẽ đối chiếu dữ liệu được tạo ra trong một khoảng thời gian cụ thể (ví dụ: quét dữ liệu 1 giờ trước).
  • Cơ chế tự động sửa lỗi (Auto-Healing):
    • Nếu bản ghi có trong MongoDB nhưng thiếu trong PostgreSQL: Thực hiện ghi bù (Insert) vào PostgreSQL.
    • Nếu bản ghi có ở PostgreSQL nhưng thiếu ở MongoDB (do lỗi đồng bộ ngược): Thực hiện ghi bù vào MongoDB (tùy thuộc vào việc bên nào được chọn là nguồn chân lý tại thời điểm đó).
    • Nếu bản ghi lệch các trường dữ liệu phụ (như ngày tạo): Thực hiện cập nhật (Update) để đồng bộ thông tin.
  • Báo cáo giám sát: Báo cáo số lượng lệch ra hệ thống giám sát (Prometheus/Grafana) để theo dõi chất lượng đồng bộ của hệ thống.

Mã cấu trúc của Reconciliation Job trong Java:

RelationReconciliationJob.java Quét đối soát và tự động sửa lỗi dữ liệu 
@Component
@Slf4j
public class RelationReconciliationJob {
    @Autowired
    private MongoRelationRepository mongoRepo;
    @Autowired
    private PostgresRelationRepository pgRepo;

    // Chạy đối soát mỗi 5 phút một lần để sửa lỗi dữ liệu 1 giờ trước
    @Scheduled(cron = "0 */5 * * * *")
    public void reconcileUserRelations() {
        Date oneHourAgo = new Date(System.currentTimeMillis() - 3600000);
        Date fiveMinutesAgo = new Date(System.currentTimeMillis() - 300000);

        // 1. Quét dữ liệu trong khung thời gian từ MongoDB
        List<MongoRelation> mongoRecords = mongoRepo.findByCreatedAtBetween(oneHourAgo, fiveMinutesAgo);

        for (MongoRelation mongoRecord : mongoRecords) {
            long follower = Long.parseLong(mongoRecord.getFollowerId());
            long following = Long.parseLong(mongoRecord.getFollowingId());

            // 2. Kiểm tra sự tồn tại bên PostgreSQL
            boolean existsInPg = pgRepo.existsByFollowerIdAndFollowingId(follower, following);

            if (!existsInPg) {
                log.warn("[RECONCILE-HEAL] Found record missing in Postgres for follower: {}, following: {}. Healing now...", follower, following);
                
                // Sửa lỗi bằng cách ghi bù dữ liệu thiếu
                PgRelation pgRelation = new PgRelation(follower, following);
                pgRepo.save(pgRelation);
            }
        }
    }
}
