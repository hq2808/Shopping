1. Luồng Upload Asset lên Google Cloud Storage (GCS)

GCP Java SDK

Sinh link upload pre-signed trực tiếp từ Client lên bucket để giảm tải băng thông Web Server, đồng thời lắng nghe Pub/Sub để xác nhận tài nguyên:

GcsAssetService.java Java 
@Service
public class GcsAssetService {
    private final Storage storage; // Google Cloud Storage client
    private final String bucketName = "main-assets-bucket";

    @Autowired
    public GcsAssetService(Storage storage) {
        this.storage = storage;
    }

    public URL generateUploadPresignedUrl(String objectName, String contentType) {
        BlobInfo blobInfo = BlobInfo.newBuilder(bucketName, objectName)
                                    .setContentType(contentType)
                                    .build();
        
        // Link upload có giá trị trong vòng 15 phút sử dụng phương thức PUT
        return storage.signUrl(blobInfo, 15, TimeUnit.MINUTES,
            Storage.SignUrlOption.httpMethod(HttpMethod.PUT),
            Storage.SignUrlOption.withExtHeaders(Map.of("Content-Type", contentType)),
            Storage.SignUrlOption.withV4Signature()
        );
    }
}

2. Tích hợp cổng thanh toán So-net (Checkout & Callback)

Payment Flow

Xử lý chuyển hướng an toàn và callback xác nhận đơn hàng từ cổng thanh toán trong Java sử dụng WebClient:

CheckoutController.java Java 
@RestController
@RequestMapping("/api/shop")
public class CheckoutController {
    private final WebClient webClient;
    private final OrderRepository orderRepository;

    @PostMapping("/checkout/complete")
    public ResponseEntity<Void> completePayment(@RequestBody PaymentCallbackDto callbackData) {
        // 1. Xác thực tính hợp lệ của chữ ký (signature verification) từ cổng So-net
        if (!verifySignature(callbackData)) {
            return ResponseEntity.status(HttpStatus.BAD_REQUEST).build();
        }

        // 2. Gọi API kiểm tra trạng thái giao dịch (confirmOrder)
        Mono<PaymentConfirmResponse> responseMono = webClient.post()
            .uri("https://api.so-net.net.tw/confirm")
            .bodyValue(new ConfirmRequest(callbackData.getOrderId()))
            .retrieve()
            .bodyToMono(PaymentConfirmResponse.class);

        responseMono.subscribe(response -> {
            if ("SUCCESS".equals(response.getStatus())) {
                // 3. Kích hoạt ví điện tử, cập nhật trạng thái đơn hàng trong DB
                orderRepository.markAsPaid(callbackData.getOrderId());
            }
        });
        return ResponseEntity.ok().build();
    }
}

3. Bộ câu hỏi phỏng vấn Senior (Q&A)

Làm cách nào thiết kế Luồng hẹn giờ hoàn tiền (Escrow Refund) cho livestream đảm bảo bền vững nếu server bị crash?
Ta KHÔNG được sử dụng các bộ hẹn giờ trong bộ nhớ (như Thread.sleep hoặc Timer trong Java) vì dữ liệu sẽ mất sạch khi server restart hoặc crash.
Giải pháp Senior: Lưu trữ trạng thái mục tiêu (Goal) trong MongoDB/PostgreSQL cùng với mốc thời gian kết thúc (Expiry Time). Sử dụng kết hợp:
  1. Redis Keyspace Notifications: Ghi một key vào Redis với TTL hết hạn bằng thời điểm buổi diễn kết thúc. Lắng nghe sự kiện hết hạn (expired event) để kích hoạt logic kiểm tra: nếu ShowGoal chưa hoàn thành -> Kích hoạt hoàn tiền cho người dùng.
  2. Distributed Scheduler (Quartz / Spring Batch): Chạy định kỳ mỗi phút quét DB để tìm các Goal đã quá hạn mà chưa được xử lý để làm cơ chế dự phòng (fail-safe) nếu Redis bị mất gói tin sự kiện.
Làm sao để Scale ngang (Horizontal Scaling) hệ thống kết nối real-time WebSocket cho phòng livestream?
Vì kết nối WebSocket giữ trạng thái (stateful - duy trì kết nối TCP liên tục giữa Client và một Server instance cụ thể), khi scale ngang thành nhiều instance chạy đằng sau Load Balancer, các Client ở các instance khác nhau sẽ không thể nhận được tin nhắn của nhau.
Giải pháp: Tận dụng cơ chế Pub/Sub thông qua một Broker tập trung (như Redis Pub/Sub, RabbitMQ, hoặc Kafka). Khi một instance nhận được tin nhắn (ví dụ: người dùng tặng quà livestream), nó sẽ gửi tin nhắn đó lên kênh Pub/Sub chung. Tất cả các instance khác đăng ký kênh này sẽ nhận được tin nhắn và phát (broadcast) trực tiếp xuống các client đang kết nối với riêng chúng.
