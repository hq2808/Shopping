1. Xử lý bất đồng bộ Spring @Async & Custom Thread Pool

ThreadPoolTaskExecutor

Tránh việc sử dụng Thread Pool mặc định (SimpleAsyncTaskExecutor - tạo luồng mới cho mỗi task làm cạn kiệt tài nguyên JVM). Ta phải định nghĩa một Thread Pool có giới hạn:

AsyncConfig.java Java 
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);          // Số luồng tối thiểu luôn hoạt động
        executor.setMaxPoolSize(50);          // Số luồng tối đa khi hàng đợi đầy
        executor.setQueueCapacity(1000);       // Dung lượng hàng đợi chứa task chờ
        executor.setThreadNamePrefix("Async-");
        // Chiến lược xử lý khi Thread Pool bị quá tải (bị đầy cả luồng lẫn hàng đợi)
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}

2. Bảo vệ Scheduled Task đa instance với ShedLock

Distributed Lock

Ngăn chặn việc các instance khác nhau của cùng một service Spring Boot chạy trùng lặp một cron task gây lỗi logic hoặc spam database:

SchedulerConfig.java Java 
@Configuration
@EnableScheduling
@EnableSchedulerLock(defaultLockAtMostFor = "10m")
public class SchedulerConfig {
    // Sử dụng Redis làm nơi lưu trữ trạng thái Khóa phân tán
    @Bean
    public LockProvider lockProvider(RedisConnectionFactory connectionFactory) {
        return new RedisLockProvider(connectionFactory);
    }
}

@Component
public class AppScheduler {
    @Scheduled(cron = "0 0/5 * * * *")
    @SchedulerLock(name = "cleanupExpiredSessions", lockAtLeastFor = "1m", lockAtMostFor = "4m")
    public void run() {
        // Logic dọn dẹp các session livestream hết hạn
    }
}

3. Bộ câu hỏi phỏng vấn Senior (Q&A)

Nếu một phương thức @Async kiểu trả về void ném ra ngoại lệ (Exception), JVM sẽ xử lý thế nào? Làm sao để bắt lỗi đó?
Vì phương thức trả về void chạy bất đồng bộ trong một luồng khác, Thread xử lý chính không thể bắt được ngoại lệ bằng khối try-catch thông thường. Ngoại lệ sẽ bị nuốt và chỉ in ra log lỗi của Thread Pool.
Để bắt và xử lý triệt để, ta phải triển khai interface AsyncConfigurer và ghi đè phương thức getAsyncUncaughtExceptionHandler() để đăng ký một custom AsyncUncaughtExceptionHandler ghi nhận log lỗi tập trung (Sentry/ELK) hoặc bắn thông báo khẩn cấp.
Lựa chọn RabbitMQ hay Kafka cho hệ thống xử lý tác vụ nền? Có gì khác biệt khi so sánh với Celery?
RabbitMQ: Phù hợp cho các bài toán phân phối tác vụ truyền thống (Task Queue) giống Celery. AMQP hỗ trợ định tuyến phức tạp (Direct, Topic, Fanout), tin nhắn được xóa ngay sau khi Consumer xác nhận (ack). Thích hợp cho luồng đặt hàng, xử lý giao dịch.
Kafka: Phù hợp cho xử lý dòng dữ liệu lớn (Event Streaming) và lưu trữ log phân tán. Tin nhắn được lưu vĩnh viễn trên ổ đĩa dựa theo thời gian cấu hình (Retention Policy), Client tự quản lý vị trí đọc (Offset). Thích hợp cho luồng phân tích hành vi người dùng, tổng hợp log hệ thống.
So với Celery: Celery là một framework hoàn chỉnh trong Python tích hợp sẵn cơ chế phân tải, kết nối task (chain, chord). Trong Java, Spring không có một thư viện tương đương hoàn toàn, lập trình viên thường tự xây dựng luồng xử lý hoặc sử dụng **Spring Cloud Stream** để giao tiếp với các Message Broker.
Làm sao đảm bảo nguyên tắc "At-Least-Once" (Ít nhất một lần nhận được tin nhắn) khi làm việc với RabbitMQ trong Spring Boot?
Để đảm bảo tin nhắn không bị thất thoát ở bất kỳ giai đoạn nào:
  1. Phía Producer: Bật tính năng Publisher Confirms và Publisher Returns của Spring RabbitTemplate để lắng nghe xác nhận từ Broker là đã nhận được tin nhắn.
  2. Phía Broker (RabbitMQ): Cấu hình Queue và Message ở chế độ durable (bền vững) để dữ liệu được ghi xuống đĩa cứng, chống mất mát khi server khởi động lại.
  3. Phía Consumer: Chuyển cấu hình Acknowledge Mode từ AUTO sang MANUAL. Chỉ gửi tín hiệu basicAck về cho Broker sau khi code xử lý nghiệp vụ thành công. Nếu xảy ra lỗi, gửi basicNack với cờ requeue=true để đẩy tin nhắn vào hàng đợi xử lý lại, hoặc cấu hình chuyển sang Dead Letter Exchange (DLX) để xử lý thủ công sau đó.

4. Cấu hình Backoff & Retry trong Spring RabbitMQ / Kafka

Tránh tình trạng Consumer bị nghẽn (Spam log liên tục) khi xử lý tin nhắn lỗi. Cấu hình Exponential Backoff (Thời gian giãn cách tăng dần) và hàng đợi tin nhắn lỗi (DLQ).

5. Cơ chế xử lý tin nhắn trùng lặp (Idempotent Consumer Pattern)

Đảm bảo một tin nhắn bị gửi trùng 2 lần do lỗi mạng không làm thay đổi trạng thái DB lần 2. Cách sử dụng Redis làm cơ chế lưu trữ phân tán kiểm tra khóa trùng (Idempotent Key).

6. Tối ưu hóa Thread Pool trong ứng dụng Reactive (Spring WebFlux)

Hiểu cơ chế Non-blocking I/O của Netty Event Loop, tránh việc thực hiện các thao tác chặn (Blocking I/O như JDBC, JPA) trên Event Loop Thread.
