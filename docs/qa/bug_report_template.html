Mẫu Bug Report & Hướng dẫn viết Bug chuẩn

Một bug report chất lượng cao là cầu nối quan trọng giữa QA và Developer. Báo cáo lỗi rõ ràng, đầy đủ thông tin giúp rút ngắn thời gian tái hiện, phân tích và sửa lỗi – từ đó nâng cao chất lượng phần mềm và hiệu suất làm việc của cả team. Trang này cung cấp chuẩn phân loại severity, mẫu đầy đủ và các ví dụ thực tế trong dự án Mini Shopee.

Critical High Medium Low

⚠️ Phân loại mức độ nghiêm trọng (Severity)

Severity phản ánh mức độ ảnh hưởng kỹ thuật của lỗi đến hệ thống, khác với Priority (mức độ ưu tiên xử lý theo nghiệp vụ). Luôn đánh giá Severity dựa trên tác động thực tế, không phụ thuộc vào cảm quan.

Mức độ Nhãn Định nghĩa Ví dụ trong Mini Shopee SLA xử lý
🔴 Critical P1 Hệ thống không thể hoạt động; dữ liệu bị sai nghiêm trọng hoặc có nguy cơ mất mát dữ liệu. Ảnh hưởng đến toàn bộ người dùng. API POST /api/orders trả về 500;
Overselling xảy ra – tồn kho về âm;
Race condition khiến DB corrupt 		≤ 4 giờ
🟠 High P1-P2 Chức năng chính bị lỗi nhưng hệ thống vẫn chạy được. Người dùng không thể hoàn thành luồng nghiệp vụ cốt lõi. Không đăng nhập được;
JWT token hết hạn nhưng vẫn nhận 200 OK;
Thanh toán bị timeout 100% 		≤ 1 ngày
🟡 Medium P2-P3 Chức năng phụ bị ảnh hưởng; người dùng có thể thực hiện luồng chính bằng cách khác (workaround tồn tại). Lịch sử đơn hàng hiển thị sai thứ tự;
UI bị vỡ layout trên mobile;
Số dư ví không cập nhật realtime 		≤ 3 ngày
🟢 Low P3 Lỗi nhỏ, không ảnh hưởng chức năng. Giao diện, văn bản, màu sắc sai nhưng hệ thống vẫn dùng bình thường. Typo trong error message;
Màu nút không đúng design;
Padding/margin lệch 2px 		Sprint tiếp theo
💡 Lưu ý: Severity ≠ Priority. Một lỗi typo trên trang thanh toán có thể là Low Severity nhưng High Priority vì ảnh hưởng đến hình ảnh thương hiệu. Hãy đánh giá cả hai chiều độc lập nhau.

📂 Ví dụ Bug Report thực tế cho Mini Shopee

Ba ví dụ dưới đây minh họa cách điền bug report đúng chuẩn cho các lỗi thường gặp trong Mini Shopee, trải dài từ Critical đến Medium.

🔴 Critical BUG-ORDER-001  ·  Overselling race condition P1
🆔 Bug ID BUG-ORDER-001
📌 Tiêu đề [CRITICAL] Overselling xảy ra khi 100 user đồng thời mua 1 sản phẩm còn 5 tồn kho
📦 Module Orders
🔴 Severity / Priority Critical  /  P1
🖥️ Môi trường Ubuntu 22.04 · Docker 26.1 · Java 17 · Spring Boot 3.2.x
Commit: main@d4f9a12
🔧 Điều kiện tiên quyết Hệ thống đã khởi chạy đầy đủ (DB, Redis, App). Product ID=1 đã được set stock_quantity = 5. 100 user account đã được tạo và lấy token.
🔁 Các bước tái hiện
  1. Reset tồn kho: UPDATE products SET stock_quantity=5 WHERE id=1;
  2. Chạy load test đồng thời 100 request:
    wrk -t10 -c100 -d5s -s post_order.lua http://localhost:8080/api/orders
  3. Sau khi wrk hoàn tất, đếm số đơn thành công:
    SELECT COUNT(*) FROM orders WHERE product_id=1 AND status='CONFIRMED';
  4. Kiểm tra tồn kho còn lại:
    SELECT stock_quantity FROM products WHERE id=1;
✅ Kết quả mong đợi Chỉ 5 đơn hàng được tạo thành công. Tồn kho còn lại = 0. 95 request còn lại nhận lỗi 409 Conflict – Insufficient stock.
❌ Kết quả thực tế 7 đơn hàng được tạo thành công. Tồn kho về -2. Không có lỗi race condition nào được log.
📎 Bằng chứng Screenshot DB query kết quả · wrk output log đính kèm
📝 Ghi chú Lỗi tái hiện 100% với ≥ 50 concurrent request. Nghi vấn: thiếu pessimistic lock / SELECT FOR UPDATE trên bảng products. Có thể liên quan đến cấu hình isolation level của MySQL (mặc định REPEATABLE READ).
🟠 High BUG-AUTH-001  ·  JWT expired token vẫn được chấp nhận P1
🆔 Bug ID BUG-AUTH-001
📌 Tiêu đề [HIGH] API trả về 200 OK khi gửi JWT token đã hết hạn thay vì 401 Unauthorized
📦 Module Auth
🔴 Severity / Priority High  /  P1
🖥️ Môi trường Ubuntu 22.04 · Postman v11 · Docker 26.1
Commit: main@a8c3e55
🔧 Điều kiện tiên quyết Hệ thống đang chạy. Có tài khoản user hợp lệ. JWT expiration được cấu hình là 24 giờ (hoặc có thể test với expiry ngắn hơn).
🔁 Các bước tái hiện
  1. POST /api/auth/login với credentials hợp lệ → lấy accessToken.
  2. Decode token trên jwt.io, xác nhận trường exp.
  3. Thay đổi thời gian hệ thống lên quá thời điểm exp (hoặc chờ token hết hạn tự nhiên).
  4. Gọi GET /api/products với header Authorization: Bearer <expired_token>.
  5. Quan sát HTTP status code và response body.
✅ Kết quả mong đợi HTTP 401 Unauthorized với body: {"error": "Token expired"}
❌ Kết quả thực tế HTTP 200 OK – dữ liệu danh sách sản phẩm được trả về bình thường. Token hết hạn không bị từ chối.
📎 Bằng chứng Postman screenshot response 200 · decoded JWT payload · server log
📝 Ghi chú Nghi vấn: filter JWT trong Spring Security có thể bỏ qua kiểm tra exp claim, hoặc clock skew quá lớn. Tái hiện Always. Đây là lỗ hổng bảo mật nghiêm trọng – cần hotfix ngay.
🟡 Medium BUG-ORDER-002  ·  Số dư ví không cập nhật realtime P2
🆔 Bug ID BUG-ORDER-002
📌 Tiêu đề [MEDIUM] Số dư ví không cập nhật realtime sau khi đặt hàng qua luồng async – phải refresh thủ công sau vài giây
📦 Module Orders Users
🔴 Severity / Priority Medium  /  P2
🖥️ Môi trường Ubuntu 22.04 · Postman v11 · Docker 26.1
Commit: main@f2b8c77
🔧 Điều kiện tiên quyết User đã đăng nhập, có đủ số dư ví để mua hàng. Endpoint async order đã được bật (POST /api/orders/async).
🔁 Các bước tái hiện
  1. GET /api/users/profile → ghi nhận walletBalance ban đầu (ví dụ: 500,000 VND).
  2. POST /api/orders/async với body hợp lệ, nhận HTTP 202 Accepted.
  3. Ngay lập tức (trong vòng 1 giây) GET /api/users/profile lần nữa.
  4. So sánh walletBalance trước và sau.
✅ Kết quả mong đợi walletBalance đã được trừ đúng số tiền của đơn hàng ngay sau khi nhận 202 Accepted (hoặc có thông báo rõ "đang xử lý").
❌ Kết quả thực tế walletBalance vẫn giữ nguyên giá trị cũ (500,000 VND). Sau khoảng 3–5 giây refresh mới thấy số dư giảm. Không có UI indicator nào thông báo trạng thái "đang xử lý".
📎 Bằng chứng Video screen recording 10 giây · Postman timeline screenshot
📝 Ghi chú Đây là hành vi expected với kiến trúc async/eventual consistency, tuy nhiên cần UX rõ ràng hơn (loading state, optimistic update, hoặc cảnh báo "số dư sẽ cập nhật sau vài giây"). Workaround: reload trang sau 5 giây. Tái hiện: Always.

✅ Checklist trước khi submit Bug

Trước khi nhấn "Submit", hãy chạy qua checklist này để đảm bảo bug report của bạn hoàn chỉnh, rõ ràng và có giá trị cao nhất với team.

  • ✅ Đã tái hiện lỗi ít nhất 2 lần độc lập – xác nhận lỗi không phải do môi trường nhất thời (flaky) hoặc thao tác nhầm.
  • ✅ Tiêu đề mô tả rõ ràng theo cấu trúc "[Severity] Ai / Làm gì / Ở đâu / Kết quả gì" – Dev đọc tiêu đề là hiểu được ngay vấn đề.
  • ✅ Đã đính kèm bằng chứng (screenshot, video hoặc log) – không có bằng chứng thì bug report gần như vô nghĩa với Developer.
  • ✅ Đã ghi rõ môi trường chạy test: OS, browser/tool version, commit hash, Docker version – đảm bảo Dev có thể setup môi trường giống hệt.
  • ✅ Severity và Priority đã được đánh giá đúng – đối chiếu với bảng phân loại ở trên trước khi chọn; tránh để mặc định hoặc chọn sai khiến triage bị lệch.
  • ✅ Các bước tái hiện đủ chi tiết và theo thứ tự – một người khác chưa biết về bug này có thể tự tái hiện mà không cần hỏi thêm.
  • ✅ Đã kiểm tra trùng lặp – tìm trong backlog/issue tracker để chắc chắn bug này chưa được report trước đó; nếu trùng thì comment thêm thông tin vào bug cũ thay vì tạo mới.
💜 Pro tip: Một bug report tốt = Dev fix nhanh hơn = QA verify sớm hơn = release đúng hạn hơn. Đầu tư 10 phút viết bug report chất lượng cao sẽ tiết kiệm hàng giờ back-and-forth giữa các bên. Chất lượng bug report là thước đo trình độ QA!
