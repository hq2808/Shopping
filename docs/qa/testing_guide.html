Tài Liệu Hướng Dẫn Kiểm Thử
(Tester Manual Guide)

Tài liệu hướng dẫn chi tiết dành cho Tester để thực hiện bắn API qua Postman và kiểm chứng các kịch bản hoạt động của hệ thống.

1. Chuẩn bị môi trường

Thông tin môi trường & Khởi chạy

Địa chỉ cổng API mặc định của máy chủ Backend cục bộ: http://localhost:8080

Trước khi kiểm thử, Tester yêu cầu lập trình viên chạy cụm Docker Compose: docker compose up --build -d.

Tải nhanh Bộ kịch bản Postman (Postman Collection):

Để giúp việc kiểm thử nhanh gọn và tự động nhất (không cần sao chép Token thủ công), bạn có thể tải bộ cấu hình Postman tích hợp sẵn kịch bản tự động lưu Token:

Tải Xuống Postman Collection JSON ↓
  • Cơ chế tự động: Khi bạn gọi API User Login hoặc Admin Login, script sẽ tự động bóc tách JWT từ phản hồi và lưu vào biến jwt_token.
  • Kế thừa: Tất cả các API còn lại trong bộ (Đặt hàng, Xem lịch sử, CRUD...) đã được cấu hình kế thừa Bearer Token từ biến này để chạy ngay mà không cần cấu hình thủ công.

Kết nối trực tiếp Cơ sở dữ liệu (Direct Database Connection)

Tester có thể sử dụng các công cụ quản lý DB (như DBeaver, pgAdmin, hoặc Navicat) để kết nối trực tiếp vào cơ sở dữ liệu PostgreSQL local nhằm kiểm tra dữ liệu thực tế lưu trữ trong bảng:

Thông số Giá trị kết nối Lưu ý kiểm thử DB
Database Type PostgreSQL Chọn Driver PostgreSQL trong công cụ DB client.
Host localhost (Hoặc địa chỉ IP máy của Developer nếu kết nối từ thiết bị mạng LAN khác).
Port 5432 Cổng tiêu chuẩn đã expose qua Docker Compose.
Database Name shopee_mini Cơ sở dữ liệu chính của dự án.
Username sa_user Tài khoản truy cập DB.
Password sa_password Mật khẩu truy cập DB.
Mục tiêu xác thực DB cho Tester:
  • Bảng users: Xác minh mật khẩu đăng ký được băm (hash) dạng BCrypt và kiểm tra số dư ví balance bị trừ chính xác sau khi đặt hàng.
  • Bảng products: Kiểm tra lượng tồn kho stock cập nhật tức thời khi mua thành công, và đảm bảo hoàn toàn không bị thay đổi nếu đơn hàng bị lỗi (kiểm tra rollback).
  • Bảng orders & order_items: Xác minh trạng thái đơn hàng và giá trị tiền của đơn hàng được chốt chính xác tại thời điểm mua.
Kịch bản kiểm thử 1

Quản Lý Sản Phẩm (Product Management CRUD)

Nhóm kịch bản kiểm thử các tính năng truy vấn, tạo mới, chỉnh sửa và xóa sản phẩm không cần và có cần vai trò quản trị.

Kịch bản 1.1: Lấy danh sách tất cả sản phẩm (Public API)

  • Method: GET
  • URL: http://localhost:8080/api/products
  • Kết quả API mong đợi: HTTP 200 OK kèm JSON danh sách sản phẩm phân trang.

Kịch bản 1.2: Lấy chi tiết sản phẩm và Xem sản phẩm lỗi

  • Đúng cú pháp: GET http://localhost:8080/api/products/1 (200 OK)
  • Không tồn tại: GET http://localhost:8080/api/products/999 (404 Not Found)

Kịch bản 1.3: Biên dữ liệu lỗi khi thêm sản phẩm (Validation)

Đăng nhập tài khoản admin để lấy Token, dán vào cấu hình Authorization. Thực hiện thêm mới sản phẩm sai định dạng:

  • Method: POST
  • URL: http://localhost:8080/api/products
  • Body (JSON): 
    {
    "name": "",
    "price": -120.00,
    "stock": -5
}
  • Kết quả API mong đợi (400 Bad Request): Lỗi validation trường `name`, `price`, `stock`.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Chạy câu lệnh SQL để đảm bảo sản phẩm lỗi không hề được lưu xuống DB: 
    SELECT * FROM products WHERE price < 0 OR stock < 0 OR name = '';
    Kết quả mong đợi: Không trả về bất kỳ dòng dữ liệu nào (0 rows).

Kịch bản 1.4: Thêm sản phẩm thành công (Admin)

Gửi yêu cầu thêm sản phẩm MacBook Pro hợp lệ:

  • Method: POST
  • URL: http://localhost:8080/api/products
  • Body (JSON): 
    {
    "name": "MacBook Pro M3",
    "description": "Apple MacBook Pro M3 14 inch",
    "price": 2000.00,
    "stock": 5
}
  • Kết quả API mong đợi: HTTP 201 Created kèm JSON thông tin sản phẩm mới vừa được sinh ID.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Chạy truy vấn SQL kiểm chứng ghi nhận chính xác dữ liệu sản phẩm mới: 
    SELECT id, name, price, stock FROM products WHERE name = 'MacBook Pro M3';
    Kết quả mong đợi: Trả về đúng 1 bản ghi với đơn giá 2000.00 và tồn kho 5.

Kịch bản 1.5: Cập nhật sản phẩm thành công (Admin)

Gửi yêu cầu chỉnh sửa sản phẩm vừa thêm ở Kịch bản 1.4 (giả sử ID = 4):

  • Method: PUT
  • URL: http://localhost:8080/api/products/4
  • Body (JSON): 
    {
    "name": "MacBook Pro M3 Max",
    "description": "Apple MacBook Pro M3 Max 16 inch",
    "price": 3500.00,
    "stock": 3
}
  • Kết quả API mong đợi: HTTP 200 OK kèm JSON thông tin sản phẩm đã chỉnh sửa.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Chạy truy vấn SQL kiểm tra cập nhật DB: 
    SELECT name, price, stock FROM products WHERE id = 4;
    Kết quả mong đợi: Bản ghi được cập nhật sang tên mới MacBook Pro M3 Max, đơn giá 3500.00 và kho 3.

Kịch bản 1.6: Xóa sản phẩm chưa có đơn hàng thành công (Admin)

Thực hiện xóa sản phẩm MacBook Pro M3 Max (ID = 4) chưa phát sinh đơn hàng nào:

  • Method: DELETE
  • URL: http://localhost:8080/api/products/4
  • Kết quả API mong đợi: HTTP 204 No Content (Xóa thành công không có nội dung trả về).
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Truy vấn xem sản phẩm còn trong DB hay không: 
    SELECT count(*) FROM products WHERE id = 4;
    Kết quả mong đợi: Trả về 0 (Đã bị xóa sạch).

Kịch bản 1.7: Xóa sản phẩm ĐÃ CÓ đơn hàng (Admin - Thất bại do ràng buộc DB)

Thử xóa sản phẩm iPhone 15 Pro (ID = 1) đã từng được đặt mua trong hệ thống:

  • Method: DELETE
  • URL: http://localhost:8080/api/products/1
  • Kết quả API mong đợi: HTTP 500 Internal Server Error do xung đột khóa ngoại từ cơ sở dữ liệu PostgreSQL.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Đảm bảo PostgreSQL từ chối xóa để giữ vững toàn vẹn tham chiếu lịch sử mua hàng: 
    SELECT count(*) FROM products WHERE id = 1;
    Kết quả mong đợi: Trả về 1. Sản phẩm vẫn tồn tại bình thường và không bị ảnh hưởng.
Kịch bản kiểm thử 2

Xác Thực Đăng Ký & Đăng Nhập (JWT Authentication)

Nhóm kịch bản kiểm tra an toàn bảo mật khi tạo tài khoản và sinh chuỗi Token JWT xác thực.

Kịch bản 2.1: Đăng ký thành công tài khoản kiểm thử

  • Method: POST
  • URL: http://localhost:8080/api/auth/register
  • Body (JSON) hỗ trợ nạp tiền ví để test: 
    {
    "username": "tester_wife",
    "password": "mypassword",
    "email": "wife@tester.com",
    "role": "ROLE_USER",
    "initialBalance": 2000.00
}
  • Kết quả API mong đợi (201 Created): Trả về thông tin User mới thành công.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Kiểm tra độ an toàn mật khẩu và số dư ví trong DB: 
    SELECT username, password, email, role, balance FROM users WHERE username = 'tester_wife';
    Kết quả mong đợi:
    • Cột password phải hiển thị chuỗi băm bảo mật BCrypt (bắt đầu bằng ký tự $2a$), **tuyệt đối không được hiển thị chữ gốc mypassword**.
    • Cột balance hiển thị đúng số dư ví 2000.00.

Kịch bản 2.2: Đăng ký trùng tên tài khoản

  • Gửi lại body đăng ký của `tester_wife` một lần nữa.
  • Kết quả API mong đợi: HTTP 400 Bad Request thông báo: "Username is already taken".
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Kiểm tra trùng lặp khóa duy nhất trong DB: 
    SELECT count(*) FROM users WHERE username = 'tester_wife';
    Kết quả mong đợi: Trả về đúng giá trị là 1. DB chặn không cho tạo trùng lặp dữ liệu.

Kịch bản 2.3: Xem thông tin Profile người dùng (Authenticated API)

Đăng nhập tài khoản bất kỳ (Ví dụ: user), sử dụng Token gán vào Authorization Bearer để gửi yêu cầu lấy profile:

  • Method: GET
  • URL: http://localhost:8080/api/users/profile
  • Kết quả API mong đợi (200 OK): Trả về thông tin cá nhân kèm số dư ví thời gian thực.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Đối soát tiền ví hiển thị ở API với ví lưu trữ thực tế trong DB: 
    SELECT balance FROM users WHERE username = 'user';
    Kết quả mong đợi: Số dư trả về khớp hoàn toàn với số dư hiển thị trên JSON Profile API.
Mẹo phân tích Token JWT:
Sau khi đăng nhập thành công qua POST /api/auth/login, Tester copy chuỗi token nhận được đem dán vào trang web jwt.io để phân tích xem có đúng thông tin role và sub (tên đăng nhập) không.
Kịch bản kiểm thử 3

Phân Quyền Theo Vai Trò (Role-Based Access Control)

Nhóm kịch bản xác minh hệ thống ngăn chặn hành vi xâm nhập trái phép hoặc sai vai trò truy cập API.

Kịch bản 3.1: Gọi API được bảo vệ không truyền Authorization Header

  • API: GET http://localhost:8080/api/orders
  • Kết quả API mong đợi: Bị chặn ở tầng lọc bảo mật, trả về HTTP 403 hoặc 401.

Kịch bản 3.2: Dùng tài khoản USER gọi API của ADMIN để xóa sản phẩm (IDIOT Test)

  • Bước 1: Đăng nhập tài khoản user lấy token ROLE_USER.
  • Bước 2: Dán token vào Authorization, gửi yêu cầu xóa sản phẩm: DELETE http://localhost:8080/api/products/1.
  • Kết quả API mong đợi: HTTP 403 Forbidden báo truy cập bị từ chối.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Kiểm tra xem sản phẩm ID = 1 có bị xóa khỏi DB hay không: 
    SELECT count(*) FROM products WHERE id = 1;
    Kết quả mong đợi: Trả về giá trị là 1. Sản phẩm vẫn nguyên vẹn trong DB, chứng minh phân quyền chặn thành công ở tầng ứng dụng và DB.

Kịch bản 3.3: Dùng tài khoản ADMIN gọi API đặt hàng của USER

  • Bước 1: Đăng nhập tài khoản admin lấy token ROLE_ADMIN.
  • Bước 2: Gửi yêu cầu đặt hàng: POST http://localhost:8080/api/orders.
  • Kết quả API mong đợi: HTTP 403 Forbidden báo truy cập bị từ chối, do tài khoản Admin không có quyền đặt hàng.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Đảm bảo không có đơn hàng nào được sinh ra cho Admin trong DB: 
    SELECT count(*) FROM orders WHERE user_id = (SELECT id FROM users WHERE username = 'admin');
    Kết quả mong đợi: Trả về 0.
Kịch bản kiểm thử 4

Quy Trình Đặt Hàng & Giao Dịch (@Transactional Rollback)

Nhóm kịch bản quan trọng nhất kiểm chứng tính chính xác của ví tiền, kho bãi và tính toàn vẹn dữ liệu trong cơ sở dữ liệu.

Kịch bản 4.1: Đặt đơn hàng thành công và Trừ tiền ví (Giao dịch Đồng bộ)

Đăng nhập tài khoản user (Ví có 1000.00 USD) và đặt mua 1 chiếc iPad Air (ID: 2, Giá 600.00 USD):

  • Method: POST
  • URL: http://localhost:8080/api/orders
  • Body (JSON): 
    {
    "items": [
        {
            "productId": 2,
            "quantity": 1
        }
    ]
}
  • Kết quả API mong đợi (201 Created): * Trả về thông tin hóa đơn với trạng thái PAID. * Ví tiền còn lại remainingBalance chính xác bằng 400.00 USD.
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Chạy đồng thời các câu lệnh SQL sau để kiểm chứng tính toàn vẹn dữ liệu: 
    -- 1. Kiểm tra ví tiền của khách hàng bị trừ chính xác 600.00 USD
SELECT balance FROM users WHERE username = 'user'; -- Mong đợi: 400.00

-- 2. Kiểm tra tồn kho iPad Air giảm từ 5 xuống 4 chiếc
SELECT stock FROM products WHERE id = 2; -- Mong đợi: 4

-- 3. Kiểm tra hóa đơn tổng được lưu chính xác trong bảng orders
SELECT id, total_amount, status FROM orders WHERE user_id = (SELECT id FROM users WHERE username = 'user') ORDER BY id DESC LIMIT 1;
-- Mong đợi: total_amount = 600.00, status = 'PAID'

-- 4. Kiểm tra dòng chi tiết đơn hàng lưu chính xác trong bảng order_items
SELECT product_id, quantity, price FROM order_items WHERE order_id = (SELECT id FROM orders WHERE user_id = (SELECT id FROM users WHERE username = 'user') ORDER BY id DESC LIMIT 1);
-- Mong đợi: product_id = 2, quantity = 1, price = 600.00

Kịch bản 4.2: Đặt đơn hàng vượt số dư ví (Giao dịch thất bại)

  • Khách hàng `user` (ví còn 400.00) gửi yêu cầu mua iPhone 15 Pro (ID: 1, Giá 1200.00 USD).
  • Kết quả API mong đợi (400 Bad Request): "Insufficient balance. Order total: 1200.00, your balance: 400.00".
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Xác minh ví tiền và kho hàng không hề bị trừ lệch: 
    -- 1. Ví tiền của user phải giữ nguyên là 400.00 USD (không bị trừ)
SELECT balance FROM users WHERE username = 'user'; -- Mong đợi: 400.00

-- 2. Tồn kho iPhone 15 Pro giữ nguyên là 10 chiếc (không bị giảm)
SELECT stock FROM products WHERE id = 1; -- Mong đợi: 10

-- 3. Không có đơn hàng mới nào được sinh ra trong DB
SELECT count(*) FROM orders WHERE total_amount = 1200.00 AND user_id = (SELECT id FROM users WHERE username = 'user');
-- Mong đợi: 0

Kịch bản 4.3: Đặt đơn hàng vượt quá tồn kho

  • Gửi yêu cầu mua sản phẩm iPad Air (ID: 2) với số lượng là 5 (trong khi kho chỉ còn 4 chiếc).
  • Kết quả API mong đợi (400 Bad Request): "Product 'iPad Air' is out of stock...".
  • Xác thực Cơ sở Dữ liệu (DB Verification):
    Đảm bảo kho và tiền không thay đổi: 
    SELECT stock FROM products WHERE id = 2; -- Mong đợi: 4
SELECT balance FROM users WHERE username = 'user'; -- Mong đợi: 400.00

Kịch bản 4.4: Kiểm thử rollback dữ liệu giao dịch cực hạn (Atomicity)

Đăng nhập tài khoản tester_wife (Ví có 2000.00 USD). Đặt đơn hàng gồm 2 sản phẩm: 2 chiếc AirPods Pro (ID: 3, Giá 250.00, kho còn 20) và 5 chiếc iPad Air (ID: 2, Giá 600.00, kho chỉ còn 4 chiếc -> Bị lỗi kho):

  • Body (JSON): 
    {
    "items": [
        {
            "productId": 3,
            "quantity": 2
        },
        {
            "productId": 2,
            "quantity": 5
        }
    ]
}
  • Bước 1: Gửi request đặt hàng. Kết quả mong đợi là hệ thống trả về lỗi 400 Bad Request báo iPad Air hết hàng.
  • Bước 2 (Xác minh tính nguyên tử trong CSDL): Chạy các câu lệnh SQL đối soát: 
    -- 1. Tồn kho của sản phẩm AirPods Pro (ID = 3) phải giữ nguyên là 20 chiếc (không được giảm đi 2)
SELECT stock FROM products WHERE id = 3; -- Mong đợi: 20

-- 2. Tồn kho của sản phẩm iPad Air (ID = 2) phải giữ nguyên là 4 chiếc
SELECT stock FROM products WHERE id = 2; -- Mong đợi: 4

-- 3. Số dư ví người dùng vẫn nguyên vẹn là 2000.00 USD
SELECT balance FROM users WHERE username = 'tester_wife'; -- Mong đợi: 2000.00

-- 4. Không có bất kỳ hóa đơn nào được tạo trong DB
SELECT count(*) FROM orders WHERE user_id = (SELECT id FROM users WHERE username = 'tester_wife'); -- Mong đợi: 0
    KẾT QUẢ ĐẠT YÊU CẦU Toàn bộ giao dịch đã được rollback thành công hoàn tất trong DB, chứng minh tính nguyên tử của `@Transactional`.
Kịch bản kiểm thử 5

Quy Trình Đặt Hàng Bất Đồng Bộ (Asynchronous Order - Redis + RabbitMQ)

Nhóm kịch bản kiểm chứng tính chính xác của ví tiền, kho bãi khi chạy qua luồng giảm chấn chịu tải cao sử dụng Redis Lua Script và RabbitMQ Message Queue.

Kịch bản 5.1: Đặt hàng bất đồng bộ thành công

Đăng nhập tài khoản user (Ví đang có 400.00 USD sau khi kết thúc kịch bản 4.1) và đặt mua 1 chiếc AirPods Pro (ID: 3, Giá 250.00 USD, kho đang còn 20 chiếc):

  • Method: POST
  • URL: http://localhost:8080/api/orders/async
  • Body (JSON): 
    {
    "items": [
        {
            "productId": 3,
            "quantity": 1
        }
    ]
}
  • Kết quả API mong đợi (202 Accepted): * Trả về HTTP Status 202 Accepted. * JSON chứa thông tin hóa đơn với orderId = null (do đơn hàng chưa ghi vào DB) và trạng thái đơn hàng là PENDING. * Số dư ví còn lại remainingBalance hiển thị là 150.00 USD.
  • Xác thực Bộ nhớ đệm (Redis Verification - Kiểm tra ngay lập tức):
    Do luồng API trừ tiền và kho ngay trên Redis RAM, hãy kết nối Redis Client (hoặc qua CLI) kiểm tra các khóa sau: 
    -- 1. Ví tiền trên Redis của user phải bị trừ ngay 250.00 USD còn 150.00 USD
GET user:balance:user -- Mong đợi: "150.00"

-- 2. Tồn kho AirPods Pro trên Redis phải giảm ngay từ 20 xuống 19 chiếc
GET product:stock:3 -- Mong đợi: "19"
  • Xác thực Cơ sở Dữ liệu (DB Verification - Chờ từ 1-2 giây để MQ tiêu thụ):
    Sau khi RabbitMQ Consumer tiêu thụ gói tin và lưu xuống DB PostgreSQL thành công: 
    -- 1. Ví tiền user trong DB được cập nhật khớp với Redis
SELECT balance FROM users WHERE username = 'user'; -- Mong đợi: 150.00

-- 2. Kho sản phẩm trong DB được cập nhật khớp với Redis
SELECT stock FROM products WHERE id = 3; -- Mong đợi: 19

-- 3. Hóa đơn được lưu trong DB với trace_id ngẫu nhiên được sinh ra
SELECT total_amount, status, trace_id FROM orders WHERE user_id = (SELECT id FROM users WHERE username = 'user') ORDER BY id DESC LIMIT 1;
-- Mong đợi: total_amount = 250.00, status = 'PAID', trace_id không null.
