Test Data Management

Hướng dẫn quản lý Test Data & Seed Data

Tài liệu này mô tả toàn bộ dữ liệu mẫu (seed data) được nạp sẵn vào hệ thống để phục vụ kiểm thử, cách reset về trạng thái ban đầu sau mỗi test run, và cách chuẩn bị dữ liệu cho từng kịch bản cụ thể. Mục tiêu là đảm bảo mọi lần chạy test đều nhất quán và lặp lại được (consistent & repeatable) mà không phụ thuộc vào trạng thái môi trường trước đó.

📋 Mục lục

  • 1. Tài khoản mặc định (Seed Users)
  • 2. Sản phẩm mặc định (Seed Products)
  • 3. Reset dữ liệu về trạng thái ban đầu
  • 4. Dữ liệu test theo kịch bản
  • 5. Câu lệnh xác minh dữ liệu sau test

1. Tài khoản mặc định Seed Users

Các tài khoản dưới đây được nạp tự động khi chạy script khởi tạo cơ sở dữ liệu init-test-db.sql. Tester có thể sử dụng ngay mà không cần tạo thêm tài khoản.

Username Password Role Balance (ví) Mục đích sử dụng
user password ROLE_USER 1,000,000đ Tester chính – dùng để đặt hàng, mua sản phẩm thông thường
admin password ROLE_ADMIN N/A Kiểm thử phân quyền admin, quản lý sản phẩm & đơn hàng
user2 password ROLE_USER 0đ Kiểm thử trường hợp số dư ví bằng 0 – đặt hàng phải bị từ chối
⚠️ Lưu ý: Các tài khoản này chỉ tồn tại sau khi chạy init-test-db.sql. Nếu bạn mới reset database hoặc chạy môi trường lần đầu, hãy chắc chắn script đã được thực thi trước khi bắt đầu bất kỳ test case nào.

2. Sản phẩm mặc định Seed Products

Bốn sản phẩm mẫu bao phủ các kịch bản phổ biến: sản phẩm giá cao, sản phẩm phổ thông, sản phẩm ít hàng và phụ kiện nhiều hàng. Tồn kho được nạp theo giá trị mặc định bên dưới.

ID Tên sản phẩm Giá Tồn kho mặc định Ghi chú
1 iPhone 15 Pro 30,000,000đ 100 Sản phẩm giá cao – test thanh toán lớn
2 Samsung Galaxy S24 20,000,000đ 50 Sản phẩm phổ thông – dùng cho flow mua hàng cơ bản
3 MacBook Pro M3 50,000,000đ 10 Sản phẩm giá cao, ít hàng – test tình huống gần hết tồn kho
4 AirPods Pro 6,000,000đ 200 Sản phẩm phụ kiện – tồn kho lớn, test mua số lượng nhiều

3. Reset dữ liệu về trạng thái ban đầu

Sau mỗi test run, dữ liệu (tồn kho, số dư ví, đơn hàng) sẽ thay đổi. Tester cần reset về trạng thái ban đầu trước khi chạy bộ test tiếp theo. Dưới đây là các lệnh reset cho từng thành phần hệ thống.

3a Reset toàn bộ bằng SQL

Chạy trực tiếp trên MySQL/MariaDB (hoặc qua DBeaver / MySQL Workbench):

-- Reset tồn kho về giá trị ban đầu
UPDATE products SET stock = 100 WHERE id = 1;
UPDATE products SET stock = 50  WHERE id = 2;
UPDATE products SET stock = 10  WHERE id = 3;
UPDATE products SET stock = 200 WHERE id = 4;

-- Reset số dư ví người dùng
UPDATE users SET balance = 1000000 WHERE username = 'user';
UPDATE users SET balance = 0       WHERE username = 'user2';

-- Xóa toàn bộ đơn hàng test
DELETE FROM order_items;
DELETE FROM orders;

3b Reset Redis Cache

Kết nối vào Redis container và ghi đè lại các key đang cache:

# Kết nối vào Redis container
docker exec -it shopee_mini_redis redis-cli

# Reset tồn kho từng sản phẩm
SET product:stock:1 100
SET product:stock:2 50
SET product:stock:3 10
SET product:stock:4 200

# Reset số dư ví
SET user:balance:user  1000000
SET user:balance:user2 0

# Hoặc xóa sạch toàn bộ cache (cẩn thận!)
FLUSHALL
⚠️ Cảnh báo: FLUSHALL xóa sạch toàn bộ dữ liệu trong Redis. Ứng dụng sẽ tự tải lại dữ liệu từ DB khi nhận request tiếp theo, nhưng có thể gây spike tải bất ngờ lên database trong môi trường production. Chỉ dùng trong môi trường test.

3c Reset RabbitMQ Queue

Xóa sạch tin nhắn tồn đọng trong queue để đảm bảo các message từ test run cũ không bị xử lý sang test run mới: 

# Xóa toàn bộ tin nhắn trong queue chính
docker exec shopee_mini_rabbitmq rabbitmqctl purge_queue shopee.order.queue

# Xóa toàn bộ tin nhắn trong Dead Letter Queue
docker exec shopee_mini_rabbitmq rabbitmqctl purge_queue shopee.order.dlq

4. Dữ liệu test theo kịch bản Scenario Data

Mỗi kịch bản kiểm thử yêu cầu trạng thái dữ liệu đầu vào khác nhau. Bảng dưới đây liệt kê điều kiện cần chuẩn bị trước khi chạy từng kịch bản.

Kịch bản Chuẩn bị dữ liệu trước khi test
✅ Test mua hàng thành công stock = 5 (ít nhất ≥ 1), balance = 150,000đ (đủ để mua); đảm bảo sản phẩm tồn tại và user đã đăng nhập
🔥 Test overselling (tải đồng thời) Đặt stock = 3, sau đó bắn 10+ request mua hàng đồng thời; kỳ vọng chỉ tối đa 3 đơn thành công, phần còn lại bị từ chối
💸 Test hết tiền Dùng tài khoản user2 (balance = 0) hoặc đặt balance < giá sản phẩm; kỳ vọng nhận lỗi "Insufficient balance"
📦 Test hết hàng Đặt stock = 0 cho sản phẩm muốn test (cả DB lẫn Redis cache); kỳ vọng nhận lỗi "Out of stock" ngay lập tức
🔁 Test idempotency Chuẩn bị 2 request POST đặt hàng mang cùng một trace_id; kỳ vọng chỉ 1 đơn hàng được tạo, request thứ 2 trả về kết quả của request đầu
💥 Test chaos – DB gặp sự cố Đảm bảo stock đủ hàng; sau khi gửi request đặt hàng, thực hiện docker pause <db_container> để giả lập DB sự cố giữa chừng; kiểm tra hệ thống retry và rollback đúng

5. Câu lệnh xác minh dữ liệu sau test Verification

Sau khi chạy test, tester cần kiểm tra trạng thái hệ thống để xác nhận kết quả đúng với kỳ vọng. Dưới đây là các query SQL thực dụng nhất để kiểm tra nhanh: 

-- Xem tồn kho hiện tại tất cả sản phẩm
SELECT id, name, stock FROM products ORDER BY id;

-- Xem 10 đơn hàng mới nhất
SELECT o.id, o.trace_id, o.status, o.created_at, u.username
FROM orders o
JOIN users u ON o.user_id = u.id
ORDER BY o.created_at DESC
LIMIT 10;

-- Xem số dư ví hiện tại của tất cả người dùng
SELECT username, balance FROM users ORDER BY username;

-- Đếm đơn hàng theo trạng thái (status)
SELECT status, COUNT(*) AS total
FROM orders
GROUP BY status;
💡 Mẹo: Chạy query đếm đơn hàng theo status trước và sau mỗi test case để nhanh chóng phát hiện đơn hàng bị kẹt ở trạng thái PENDING hoặc tăng bất thường ở FAILED. Kết hợp với log của RabbitMQ consumer để trace nguyên nhân gốc rễ.
