Performance QA Benchmark Baseline v1.0.0

Performance Benchmark & Baseline Kết quả

Tài liệu này thiết lập các ngưỡng hiệu năng baseline chính thức của Mini Shopee, được đo lường trên môi trường chuẩn hóa. Mục tiêu là cung cấp một điểm tham chiếu khách quan để so sánh và phát hiện regression hiệu năng qua mỗi release. Tất cả số liệu phải được tái đo và ghi lại vào bảng log ở cuối tài liệu trước mỗi lần phát hành.

⚙️ Thiết lập môi trường chạy Benchmark

Để đảm bảo tính nhất quán, benchmark phải được chạy trên cùng một cấu hình phần cứng và phần mềm. Bảng dưới đây mô tả môi trường chuẩn đã được sử dụng để thu thập số liệu baseline.

Thành phần Cấu hình test
Máy test (Host) 8 CPU cores, 16 GB RAM, SSD NVMe — Ubuntu 22.04 LTS
Docker resources Giới hạn 4 CPU, 8 GB RAM cho toàn bộ Compose stack

⚠️ Lưu ý quan trọng: Kết quả benchmark phụ thuộc vào phần cứng và cấu hình môi trường. Hãy sử dụng các con số này làm BASELINE so sánh giữa các release — tức là so sánh kết quả của release mới với kết quả baseline trên cùng một máy, cùng cấu hình. Không so sánh tuyệt đối với môi trường khác (staging, production, CI/CD runner khác).

📊 Kết quả Baseline (Điều kiện bình thường)

Số liệu dưới đây được thu thập khi hệ thống đang hoạt động bình thường: database đã có seed data (~1,000 sản phẩm, ~5,000 đơn hàng), Redis cache đã warm, không có maintenance job nào chạy đồng thời.

GET /api/products  — Read-heavy, Redis cache, không cần auth

Endpoint lấy danh sách sản phẩm — có phân trang, được cache tại Redis với TTL 5 phút. Đây là endpoint được truy cập nhiều nhất, kỳ vọng RPS rất cao nhờ cache hit rate > 90%.

Metric Kết quả đo được Ngưỡng PASS Trạng thái
RPS (Requests/sec) 2,500 req/s ≥ 1,000 req/s ✅ PASS
Avg Latency 45 ms ≤ 200 ms ✅ PASS
P95 Latency 120 ms ≤ 500 ms ✅ PASS
P99 Latency 280 ms ≤ 1,000 ms ✅ PASS
Error Rate 0% ≤ 0.1% ✅ PASS
Concurrency (wrk -c) 100 connections — Tham số test

POST /api/orders  — Đồng bộ, Pessimistic Lock, cần JWT

Endpoint tạo đơn hàng đồng bộ — sử dụng Pessimistic Lock trên PostgreSQL để đảm bảo tính toàn vẹn tồn kho. RPS thấp hơn đáng kể do mỗi request phải giữ row-lock cho đến khi transaction hoàn tất. Độ trễ phụ thuộc vào thời gian giữ lock và độ tranh chấp.

Metric Kết quả đo được Ngưỡng PASS Trạng thái
RPS (Requests/sec) 85 req/s ≥ 50 req/s ✅ PASS
Avg Latency 180 ms ≤ 500 ms ✅ PASS
P95 Latency 450 ms ≤ 1,500 ms ✅ PASS
P99 Latency 890 ms ≤ 3,000 ms ✅ PASS
Error Rate 0.2% ≤ 1% ✅ PASS
Concurrency (wrk -c) 50 connections — Tham số test
📝 Ghi chú: Error rate 0.2% đến từ một số request bị timeout khi DB connection pool bão hòa tạm thời. Đây là hành vi được chấp nhận ở mức tải cao với Pessimistic Lock. Nếu error rate > 1%, cần kiểm tra lại HikariCP maximumPoolSize và lock_timeout của PostgreSQL.

POST /api/orders/async  — Bất đồng bộ, Redis + RabbitMQ, cần JWT

Endpoint tạo đơn hàng bất đồng bộ — request được xác nhận nhanh và đẩy vào RabbitMQ queue, xử lý thực tế bởi consumer riêng. Redis được dùng để kiểm tra tồn kho tạm thời (optimistic approach). RPS cao hơn nhiều so với sync endpoint do không giữ DB connection trong suốt quá trình xử lý.

Metric Kết quả đo được Ngưỡng PASS Trạng thái
RPS (Requests/sec) 1,200 req/s ≥ 500 req/s ✅ PASS
Avg Latency 25 ms ≤ 100 ms ✅ PASS
P95 Latency 80 ms ≤ 300 ms ✅ PASS
P99 Latency 150 ms ≤ 600 ms ✅ PASS
Error Rate 0% ≤ 0.1% ✅ PASS
Concurrency (wrk -c) 200 connections — Tham số test
✅ Nhận xét: Async endpoint cho thấy hiệu năng vượt trội (~14× RPS so với sync). Đây là lựa chọn ưu tiên cho traffic cao. Cần đảm bảo consumer xử lý queue không bị lag bằng cách theo dõi messages depth trong RabbitMQ.

🖥️ Lệnh chạy Benchmark

Chạy đúng thứ tự các lệnh sau. Đảm bảo hệ thống đã được khởi động đầy đủ và đã qua giai đoạn warm-up 30 giây trước khi ghi nhận kết quả.

Bước 0 — Warm-up hệ thống

Gửi 30 giây request nhẹ để JVM JIT compile và Redis cache được làm đầy trước khi đo chính thức:

# Warm-up GET endpoint (30s, không ghi kết quả)
wrk -t2 -c20 -d30s http://localhost:8080/api/products

echo "=== Warm-up complete. Chờ 5 giây trước khi đo chính thức ==="
sleep 5

Bước 1 — Lấy JWT Token

# 1. Lấy JWT token trước
TOKEN=$(curl -s -X POST http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"user","password":"password"}' | jq -r '.token')

# Kiểm tra token hợp lệ
echo "Token: ${TOKEN:0:50}..."

Bước 2 — Benchmark GET /api/products

# 2. Benchmark GET /api/products (không cần body)
wrk -t4 -c100 -d30s \
  -H "Authorization: Bearer $TOKEN" \
  --latency http://localhost:8080/api/products

-t4 4 luồng wrk   -c100 100 kết nối đồng thời   -d30s đo trong 30 giây   --latency hiển thị phân phối latency

Bước 3 & 4 — Benchmark POST /api/orders (Sync)

# 3. Tạo file Lua script cho POST order (sync)
cat > benchmark_order_sync.lua << 'EOF'
wrk.method = "POST"
wrk.body   = '{"items":[{"productId":2,"quantity":1}]}'
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer TOKEN_HERE"
EOF

# Thay TOKEN_HERE bằng token thực
sed -i "s/TOKEN_HERE/$TOKEN/" benchmark_order_sync.lua

# 4. Benchmark POST /api/orders (sync, pessimistic lock)
wrk -t4 -c50 -d30s -s benchmark_order_sync.lua \
  --latency http://localhost:8080/api/orders

-c50 Chỉ dùng 50 connections để tránh quá tải DB lock pool. Tăng lên để test giới hạn chịu tải nhưng không dùng số cao hơn cho baseline.

Bước 5 — Benchmark POST /api/orders/async

# Tạo Lua script cho async order
cat > benchmark_order_async.lua << 'EOF'
wrk.method = "POST"
wrk.body   = '{"items":[{"productId":3,"quantity":1}]}'
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer TOKEN_HERE"
EOF

sed -i "s/TOKEN_HERE/$TOKEN/" benchmark_order_async.lua

# 5. Benchmark POST /api/orders/async
wrk -t8 -c200 -d30s -s benchmark_order_async.lua \
  --latency http://localhost:8080/api/orders/async

-t8 -c200 Dùng 8 luồng và 200 connections để mô phỏng tải cao — async endpoint được thiết kế chịu tải này.

📡 Giám sát tài nguyên trong khi chạy benchmark

Mở terminal riêng biệt để chạy các lệnh monitoring song song với benchmark. Ghi lại peak values (giá trị cao nhất) vào bảng log release.

Docker Container Stats (Realtime)

# Monitor Docker container resources realtime
# Ctrl+C để dừng
docker stats shopee_mini_app shopee_mini_db shopee_mini_redis shopee_mini_rabbitmq

Output mẫu kỳ vọng:

NAME                CPU %   MEM USAGE / LIMIT    MEM %   NET I/O
shopee_mini_app     45%     1.2GiB / 2GiB        60%     500MB / 200MB
shopee_mini_db      20%     300MiB / 1GiB        30%     50MB / 100MB
shopee_mini_redis   5%      80MiB / 512MiB       16%     200MB / 200MB
shopee_mini_rabbitmq 3%     120MiB / 512MiB      23%     150MB / 150MB

Kiểm tra DB Connections đang mở

# Check DB connections đang mở (chạy trong lúc benchmark đang chạy)
docker exec shopee_mini_db psql -U sa_user -d shopee_mini -c \
  "SELECT count(*) as active_connections FROM pg_stat_activity WHERE state = 'active';"

# Xem chi tiết từng connection (query đang chạy)
docker exec shopee_mini_db psql -U sa_user -d shopee_mini -c \
  "SELECT pid, state, wait_event_type, wait_event, query_start, query
   FROM pg_stat_activity
   WHERE state != 'idle'
   ORDER BY query_start;"

Ngưỡng cảnh báo: Active connections > 8 / 10 (80% HikariCP pool) — có thể gây connection timeout.

Kiểm tra Redis Memory Usage

# Check Redis memory usage
docker exec shopee_mini_redis redis-cli INFO memory | grep used_memory_human

# Xem cache hit rate
docker exec shopee_mini_redis redis-cli INFO stats | grep -E 'keyspace_hits|keyspace_misses'

# Tính hit rate thủ công
docker exec shopee_mini_redis redis-cli INFO stats | awk '
  /keyspace_hits/ {hits=$2}
  /keyspace_misses/ {misses=$2}
  END {print "Hit rate: " hits/(hits+misses)*100 "%"}'

Kỳ vọng: used_memory < 256 MiB, hit rate > 90% khi benchmark GET /api/products.

Kiểm tra RabbitMQ Queue Depth

# Check RabbitMQ queue depth
docker exec shopee_mini_rabbitmq rabbitmqctl list_queues name messages consumers

# Theo dõi liên tục mỗi 2 giây
watch -n 2 "docker exec shopee_mini_rabbitmq rabbitmqctl list_queues name messages consumers"

Lưu ý: Nếu messages trong queue tăng liên tục (không ổn định) trong lúc benchmark, consumer đang bị lag — cần tăng số consumer threads hoặc kiểm tra DB write bottleneck. Sau khi benchmark kết thúc, queue phải về 0 trong vòng 30 giây.

JVM Heap Monitoring (tuỳ chọn)

# Xem JVM heap usage qua actuator (nếu Spring Actuator được bật)
curl -s http://localhost:8080/actuator/metrics/jvm.memory.used | jq '.measurements[0].value / 1048576 | . * 100 | round / 100 | tostring + " MiB"'

# Hoặc dùng jstat nếu có JDK trên host
# Lấy PID của Java process trong container
PID=$(docker exec shopee_mini_app sh -c 'jps | grep -v Jps | head -1 | cut -d" " -f1')
docker exec shopee_mini_app jstat -gc $PID 2000 10

🚨 Ngưỡng Regression Alert

⚠️ Quy tắc Regression: Nếu kết quả benchmark trong release mới tệ hơn baseline từ 20% trở lên trên bất kỳ chỉ số nào được liệt kê dưới đây, bắt buộc phải điều tra và tìm root cause trước khi tiến hành release. Hãy tạo bug report theo mẫu Bug Report Template với tag Performance Regression.
Endpoint Baseline RPS 🚨 Alert nếu RPS dưới Baseline P95 🚨 Alert nếu P95 trên Ngưỡng (%)
GET /api/products 2,500 req/s < 2,000 req/s 120 ms > 150 ms −20% / +25%
POST /api/orders 85 req/s < 68 req/s 450 ms > 562 ms −20% / +25%
POST /api/orders/async 1,200 req/s < 960 req/s 80 ms > 100 ms −20% / +25%

Checklist khi phát hiện Regression

  • ✅ Xác nhận lại môi trường test giống với baseline (cùng Docker config, JVM flags)
  • ✅ Kiểm tra có migration DB nào mới làm chậm query không (EXPLAIN ANALYZE)
  • ✅ Kiểm tra có N+1 query nào mới được introduce không (enable Hibernate SQL logging)
  • ✅ Kiểm tra Redis cache TTL có bị thay đổi không
  • ✅ Kiểm tra HikariCP pool size có bị giảm không
  • ✅ So sánh GC logs trước và sau (-Xlog:gc*)
  • ✅ Kiểm tra có dependency mới nào có bug hiệu năng không
  • ✅ Tạo bug report với label Performance Regression và đính kèm wrk output
