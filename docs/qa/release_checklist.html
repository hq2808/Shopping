Tài liệu này định nghĩa toàn bộ các hạng mục kiểm thử bắt buộc phải hoàn thành và đạt (PASS) trước mỗi lần phát hành phiên bản mới của hệ thống Mini Shopee. Mọi lần deploy lên môi trường Staging hoặc Production đều phải có sign-off của QA Lead dựa trên checklist này.

Smoke Test Checklist ⏱ ~5 phút

Đây là bộ kiểm tra tối thiểu để xác nhận hệ thống đang hoạt động. Chạy đầu tiên — trước bất kỳ bước kiểm thử nào khác. Nếu bất kỳ mục nào thất bại, dừng lại ngay và thông báo cho Dev.

✅
App khởi động không có lỗi trong logs
Kiểm tra logs: port 8080 đang lắng nghe, không có ERROR hay FATAL trong 30 giây đầu tiên sau khi start.
Port 8080 OK
✅
PostgreSQL kết nối thành công
Không xuất hiện Connection refused hay HikariPool timeout trong logs. Ít nhất một pool connection được thiết lập.
DB Connected
✅
Redis kết nối thành công
Logs hiển thị Lettuce/Jedis connected to redis://localhost:6379 hoặc tương đương. Không có RedisConnectionException.
Redis OK
✅
RabbitMQ kết nối thành công
Spring AMQP log báo Successfully connected to RabbitMQ. Management UI tại localhost:15672 hiển thị connection đang active.
MQ Connected
✅
POST /api/auth/login trả về 200 OK và JWT token
Dùng credentials seed mặc định user01 / password123. Response body phải chứa field token dạng JWT (3 phần phân tách bởi dấu chấm).
200 OK + JWT
✅
GET /api/products trả về danh sách sản phẩm
Response 200 OK, body là mảng JSON có ít nhất 1 phần tử. Thời gian phản hồi < 2 giây.
200 OK + Data
🛑
ĐIỀU KIỆN DỪNG: Nếu bất kỳ mục nào trong Smoke Test FAIL → STOP ngay lập tức, không tiếp tục các bước kiểm thử tiếp theo. Tạo bug report và thông báo cho Dev Lead / Release Manager để rollback hoặc fix trước khi tiến hành lại.
🔍

Functional Regression Checklist ⏱ ~30 phút

Kiểm thử toàn bộ chức năng nghiệp vụ theo từng module. Thực hiện theo thứ tự từ trên xuống dưới. Ghi lại kết quả PASS / FAIL cho từng mục.

🔐 AUTH MODULE — Xác thực & Phân quyền
Login thành công với user/password hợp lệ → nhận JWT 200 OK
Login thất bại với password sai → bị từ chối 401 Unauthorized
Register tài khoản mới với email chưa tồn tại → thành công 201 Created
Gọi API có auth với token hết hạn → bị từ chối 401 Unauthorized
Gọi API có auth không kèm token → bị từ chối 401 Unauthorized
📦 PRODUCTS MODULE — Quản lý sản phẩm
GET /api/products — Lấy danh sách sản phẩm, có dữ liệu 200 OK
GET /api/products/{id} — Lấy chi tiết sản phẩm tồn tại, đúng data 200 OK
GET /api/products/99999 — Lấy sản phẩm không tồn tại → not found 404 Not Found
⚡ ORDERS — SYNC MODULE — Đặt hàng đồng bộ
Đặt hàng thành công (đủ tồn kho, đủ số dư ví) → đơn hàng ghi vào DB 200 OK
Đặt hàng khi sản phẩm hết tồn kho → bị từ chối với message rõ ràng 400 Bad Request
Đặt hàng khi số dư ví không đủ → bị từ chối với message rõ ràng 400 Bad Request
Xác nhận tồn kho products.stock giảm đúng số lượng sau khi đặt hàng thành công Stock OK
Xác nhận số dư ví users.balance giảm đúng giá trị đơn hàng sau khi đặt thành công Balance OK
🐇 ORDERS — ASYNC MODULE — Đặt hàng bất đồng bộ
POST /api/orders/async — Gửi yêu cầu đặt hàng → chấp nhận ngay 202 Accepted
Kiểm tra RabbitMQ Management UI / logs: tin nhắn được publish thành công lên queue Published
Consumer xử lý tin nhắn → đơn hàng xuất hiện trong DB (orders table) trong vòng 5 giây DB OK
Redis stock cache giảm đúng số lượng sau khi consumer xử lý xong Redis OK
👤 USERS MODULE — Thông tin người dùng
GET /api/users/profile với token hợp lệ → trả về thông tin đúng của user 200 OK
Profile của user A không thể xem bằng token của user B → bị từ chối 403 Forbidden
🗄️

Data Integrity Checklist ⏱ ~15 phút

Chạy trực tiếp các câu query SQL dưới đây vào PostgreSQL sau khi hoàn thành Functional Test. Đây là bước đối soát dữ liệu để đảm bảo tính toàn vẹn của hệ thống. Kết nối DB: psql -h localhost -U postgres -d shopee_db

⚠️ Lưu ý: Các query #2, #3, #4 BẮT BUỘC phải trả về 0 rows. Nếu có kết quả → hệ thống có lỗi nghiêm trọng về toàn vẹn dữ liệu → STOP release.
QUERY 1 — Tổng đơn hàng trong 1 giờ gần nhất Expected: Số đếm ≥ 0 
-- 1. Kiểm tra tổng đơn hàng khớp với giao dịch trong giờ vừa qua
SELECT COUNT(*) AS total_orders
FROM   orders
WHERE  created_at > NOW() - INTERVAL '1 hour';
QUERY 2 — Tồn kho âm (CRITICAL) Expected: 0 rows 
-- 2. Kiểm tra tồn kho không được phép âm (vi phạm business rule)
SELECT id, name, stock
FROM   products
WHERE  stock < 0;
QUERY 3 — Trùng lặp trace_id (CRITICAL) Expected: 0 rows 
-- 3. Kiểm tra không có trace_id trùng lặp (idempotency violation)
SELECT   trace_id,
         COUNT(*) AS duplicate_count
FROM     orders
GROUP BY trace_id
HAVING   COUNT(*) > 1;
QUERY 4 — Số dư ví âm (CRITICAL) Expected: 0 rows 
-- 4. Kiểm tra số dư ví không được phép âm (tài khoản không được thâm hụt)
SELECT username, balance
FROM   users
WHERE  balance < 0;
📋 Hướng dẫn đối soát bổ sung:
  • So sánh SUM(orders.total_amount) với SUM(transactions.amount) nếu có bảng giao dịch riêng
  • Kiểm tra orders.status không có giá trị NULL hoặc trạng thái không hợp lệ
  • Verify rằng mỗi order đều có user_id trỏ về user tồn tại trong bảng users
  • Kiểm tra Redis stock cache khớp với PostgreSQL products.stock
🔒

Security Checklist ⏱ ~10 phút

Kiểm thử bảo mật cơ bản — không thay thế cho penetration testing chuyên sâu, nhưng phải pass 100% trước mọi lần release.

🛡️
JWT Role Isolation — User không access Admin resource RBAC
Dùng JWT của user thường gọi endpoint admin (ví dụ: GET /api/admin/users). Phải bị từ chối với 403 Forbidden. Không được nhận 200 OK dù token hợp lệ.
403 Forbidden
🔑
JWT Tampering — Token giả mạo bị reject JWT
Lấy JWT hợp lệ, sửa payload (ví dụ: đổi role: "USER" thành role: "ADMIN"), ký lại bằng secret giả. API phải từ chối với 401 Unauthorized.
401 Unauthorized
🏛️
RBAC Enforcement — User thường không gọi được Admin endpoint RBAC
Thử gọi các endpoint chỉ dành cho ADMIN role bằng token user thường. Kiểm tra tất cả các admin route đều được bảo vệ bởi @PreAuthorize("hasRole('ADMIN')") hoặc tương đương.
403 Forbidden
💉
SQL Injection Protection — Input độc hại bị reject Injection
Gửi request với productId = "1; DROP TABLE products" hoặc productId = "1' OR '1'='1". Phải nhận lỗi validation (400/404), không được thực thi SQL, không có dữ liệu bị xóa.
400 Bad Request
🔢
Negative Amount Rejection — Số lượng âm bị từ chối Input Validation
Đặt hàng với quantity: -1 hoặc quantity: 0. API phải từ chối với lỗi validation rõ ràng — không được trừ stock, không được hoàn tiền âm vào ví.
400 Bad Request
🚨 Lưu ý bảo mật: Nếu bất kỳ security check nào FAIL → đây là blocker nghiêm trọng. Release phải bị dừng ngay lập tức và lỗ hổng phải được vá trước khi deploy.
📋

Release Sign-off

Bảng ký nhận kết quả kiểm thử cuối cùng. QA Lead phải điền đầy đủ và ký tên trước khi gửi thông báo phê duyệt release cho Release Manager và Tech Lead.

🗄️ Data Integrity _______________ PASS / FAIL __/__/____ __:__ ___________________
🔒 Security Check _______________ PASS / FAIL __/__/____ __:__ ___________________
⚡ Performance Baseline _______________ PASS / FAIL __/__/____ __:__ ___________________
✍️ Sign-off QA Lead _______________ PASS / FAIL __/__/____ __:__ ___________________
✅ Điều kiện PASS toàn bộ
  • 100% Smoke Test items PASS
  • ≥ 95% Functional items PASS (không có blocker)
  • 100% Data Integrity queries trả về 0 rows (queries 2, 3, 4)
  • 100% Security checks PASS
  • Performance không tệ hơn baseline > 20%
🚫 Điều kiện FAIL / Block release
  • Bất kỳ Smoke Test item nào FAIL
  • Bất kỳ Security check nào FAIL
  • Data Integrity query 2, 3, hoặc 4 có kết quả > 0 rows
  • Có bug P0 / Blocker chưa được fix
  • Performance tệ hơn baseline > 50%
