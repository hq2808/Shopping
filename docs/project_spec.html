Project Specifications

Tài Liệu Đặc Tả Dự Án
(Mini Shopee System Spec)

Tài liệu cung cấp thông tin kỹ thuật chuyên sâu về luồng hoạt động, cấu trúc cơ sở dữ liệu và các quy tắc nghiệp vụ giúp tester xây dựng kịch bản kiểm thử toàn diện.

1. Tổng Quan

1. Tổng quan hệ thống

Mini Shopee là một hệ thống backend thương mại điện tử tinh gọn, được phát triển phục vụ cho các bài toán kiểm thử validation dữ liệu đầu vào, kiểm thử bảo mật với JWT, kiểm thử tính toàn vẹn giao dịch và đánh giá giải pháp chịu tải cao.

Các đặc trưng chính của hệ thống:
  • Bảo mật dựa trên Token: Sử dụng chuẩn JWT (JSON Web Token) để xác thực và phân quyền theo vai trò (ROLE_USER, ROLE_ADMIN).
  • Đặt hàng truyền thống (Đồng bộ): Trừ kho và tiền trực tiếp trong một giao dịch cơ sở dữ liệu PostgreSQL có quản lý transaction.
  • Đặt hàng chịu tải (Bất đồng bộ): Sử dụng bộ nhớ đệm Redis để giữ chỗ (reserve) tài nguyên thông qua Lua Script nguyên tử, sau đó đẩy sự kiện đặt hàng vào RabbitMQ để xử lý bất đồng bộ, giúp hệ thống chịu được lượng request khổng lồ mà không sập DB.
2. Kiến Trúc

2. Kiến trúc luồng dữ liệu đặt hàng

Luồng đặt hàng Đồng bộ (Standard Order Flow)

Thích hợp cho tải lượng trung bình, ưu tiên tính nhất quán dữ liệu tức thời:

  1. Client gửi danh sách sản phẩm và số lượng mua qua API POST /api/orders kèm JWT.
  2. Hệ thống truy vấn DB PostgreSQL để kiểm tra tồn kho và đơn giá sản phẩm.
  3. Kiểm tra số dư tài khoản của User trong DB.
  4. Nếu đủ điều kiện: thực hiện trừ kho, trừ tiền trong DB và lưu hóa đơn (gói gọn trong `@Transactional` của Spring).
  5. Trả về phản hồi 201 Created kèm hóa đơn đã thanh toán.

Luồng đặt hàng Bất đồng bộ (High-Concurrency Order Flow)

Thích hợp cho các chiến dịch Sale lớn (Flash Sale), giảm tải trực tiếp cho DB:

  1. Client gửi danh sách sản phẩm qua API POST /api/orders/async kèm JWT.
  2. Backend lấy siêu dữ liệu (tên, giá) từ cache Redis (hoặc PostgreSQL nếu lỡ cache miss).
  3. Gọi Lua Script nguyên tử chạy trên RAM của Redis để kiểm tra đồng thời:
    • Số dư ví của User (Key Redis: user:balance:{username}) có đủ thanh toán tổng đơn hàng hay không.
    • Tồn kho từng sản phẩm (Key Redis: product:stock:{id}) có đủ số lượng yêu cầu hay không.
  4. Nếu Lua Script trả về thành công:
    • Kho và tiền của user trên Redis RAM sẽ được khấu trừ ngay lập tức.
    • Một thông điệp chứa thông tin đặt hàng được đẩy vào RabbitMQ queue shopee.order.queue.
    • Hệ thống lập tức trả về 202 Accepted với trạng thái đơn hàng là PENDING.
  5. OrderConsumer chạy nền lắng nghe queue RabbitMQ, tuần tự lấy thông điệp ra ghi nhận xuống PostgreSQL DB thực tế. Trạng thái đơn hàng chuyển sang PAID.
3. Database Schema

3. Đặc tả cơ sở dữ liệu (PostgreSQL)

Hệ thống lưu trữ dữ liệu dưới cấu trúc 4 bảng quan hệ chính:

Bảng users (Người dùng)

Tên trường Kiểu dữ liệu Ràng buộc Mô tả
id BIGINT Primary Key, Auto-increment Mã định danh duy nhất của người dùng.
username VARCHAR(255) Unique, Not Null Tên đăng nhập (độ dài 3-20 ký tự).
password VARCHAR(255) Not Null Mật khẩu đã được băm bằng BCrypt.
email VARCHAR(255) Unique, Not Null Địa chỉ email của người dùng.
role VARCHAR(50) Not Null Vai trò: ROLE_USER hoặc ROLE_ADMIN.
balance NUMERIC(38,2) Not Null, >= 0 Số dư ví tiền của người dùng.

Bảng products (Sản phẩm)

Tên trường Kiểu dữ liệu Ràng buộc Mô tả
id BIGINT Primary Key, Auto-increment Mã định danh sản phẩm.
name VARCHAR(255) Not Null Tên hiển thị sản phẩm.
description TEXT - Mô tả chi tiết sản phẩm.
price NUMERIC(38,2) Not Null, >= 0 Đơn giá sản phẩm.
stock INTEGER Not Null, >= 0 Số lượng sản phẩm còn lại trong kho.

Bảng orders (Đơn đặt hàng)

Tên trường Kiểu dữ liệu Ràng buộc Mô tả
id BIGINT Primary Key, Auto-increment Mã định danh đơn hàng.
trace_id VARCHAR(255) Unique, Nullable Dùng để kiểm soát trùng lặp đơn hàng bất đồng bộ.
user_id BIGINT Foreign Key (users) Liên kết đến khách hàng mua đơn này.
total_amount NUMERIC(38,2) Not Null Tổng số tiền thanh toán của toàn đơn hàng.
status VARCHAR(50) Not Null Trạng thái đơn: PENDING, PAID, CANCELLED.
created_at TIMESTAMP Not Null Thời gian khởi tạo đơn hàng.

Bảng order_items (Chi tiết đơn hàng)

Tên trường Kiểu dữ liệu Ràng buộc Mô tả
id BIGINT Primary Key, Auto-increment Mã định danh chi tiết dòng sản phẩm.
order_id BIGINT Foreign Key (orders) Liên kết đến đơn hàng tổng.
product_id BIGINT Foreign Key (products) Sản phẩm được mua.
quantity INTEGER Not Null, >= 1 Số lượng sản phẩm đặt mua.
price NUMERIC(38,2) Not Null Giá chốt sản phẩm tại thời điểm đặt hàng.
4. Quy Tắc Nghiệp Vụ

4. Quy tắc nghiệp vụ bắt buộc (Business Rules)

Tester cần nắm chắc các quy tắc hoạt động logic của hệ thống để xây dựng đầy đủ các kịch bản kiểm thử biên (Edge Cases):

4.1. Quy tắc về Giỏ hàng & Số lượng đặt hàng (Cart Constraints)

  • Giới hạn mặt hàng: Một đơn hàng chỉ được chứa tối đa 50 dòng sản phẩm khác nhau (`OrderItem`). Yêu cầu gửi danh sách rỗng hoặc vượt quá 50 sẽ nhận lỗi 400 Bad Request.
  • Độ lớn số lượng tối thiểu: Số lượng (`quantity`) đặt mua của từng sản phẩm trong giỏ hàng bắt buộc phải lớn hơn hoặc bằng 1. Yêu cầu nhỏ hơn 1 hoặc null sẽ bị chặn bởi tầng validation.

4.2. Quy tắc trừ Kho & Ví tiền (Inventory & Balance Constraints)

  • Kiểm soát tồn kho: Số lượng đặt mua của bất kỳ sản phẩm nào phải nhỏ hơn hoặc bằng số lượng tồn kho hiện tại trong hệ thống.
  • Kiểm soát khả năng chi trả: Tổng giá trị thanh toán của đơn hàng (sau khi nhân số lượng với giá chốt của từng sản phẩm) không được vượt quá số dư ví (`balance`) hiện tại của người dùng.

4.3. Tính nguyên tử giao dịch (Atomicity & Rollback)

  • Nguyên tắc Rollback: Khi đặt đơn hàng gồm nhiều sản phẩm, nếu có bất cứ dòng sản phẩm nào bị lỗi (như hết hàng) hoặc người dùng không đủ tiền thanh toán vào phút chót, hệ thống bắt buộc phải **phục hồi toàn vẹn trạng thái cũ (Rollback)**: kho của tất cả sản phẩm khác và ví tiền người dùng không được phép thay đổi.

4.4. Quy tắc bảo vệ khi xóa sản phẩm (Product Deletion Safety)

  • Ràng buộc Khóa Ngoại: Do bảng chi tiết đơn hàng (order_items) liên kết trực tiếp tới sản phẩm (products), hệ thống áp dụng ràng buộc khóa ngoại chặt chẽ. **Không cho phép xóa các sản phẩm đã từng được đặt mua**. Nếu cố tình xóa, database PostgreSQL sẽ chặn lại và trả về lỗi xung đột khóa ngoại.
⚠️ LƯU Ý LỖI NGHIỆP VỤ (VẤN ĐỀ ĐỒNG BỘ REDIS):

Trong mã nguồn hiện tại đang tồn tại một lỗ hổng đồng bộ (Synchronization Bug): Khi một user đăng ký tài khoản mới qua API POST /api/auth/register, dữ liệu được ghi xuống PostgreSQL thành công, nhưng số dư ví ban đầu không được tự động đồng bộ (sync) lên bộ nhớ đệm Redis.

  • Hậu quả: Nếu tài khoản mới tạo này thực hiện đặt hàng ngay qua luồng bất đồng bộ (POST /api/orders/async), Redis Lua Script sẽ không tìm thấy số dư ví của user và trả về lỗi -2 (Insufficient balance) mặc dù tài khoản có tiền trong DB.
  • Cách khắc phục khi test: Tester nên sử dụng tài khoản seed sẵn (user) để kiểm thử đặt hàng bất đồng bộ, hoặc sử dụng API đặt hàng đồng bộ (POST /api/orders) đối với tài khoản mới tạo.
5. API Spec

5. Đặc tả các API Endpoints

Bảng danh sách toàn bộ các API Endpoints hệ thống cung cấp:

Nhóm API Xác thực & Người dùng (Authentication & User Profile)

Method Endpoint Quyền hạn Mô tả
POST /api/auth/register Public Đăng ký tài khoản mới. Ràng buộc: Username từ 3-20 ký tự, Mật khẩu tối thiểu 6 ký tự.
POST /api/auth/login Public Đăng nhập sinh Token JWT. Trả về Token kèm thông tin username và role.
GET /api/users/profile USER / ADMIN Xem thông tin cá nhân hiện tại bao gồm số dư ví balance.

Nhóm API Quản lý sản phẩm (Products)

Method Endpoint Quyền hạn Mô tả
GET /api/products Public Lấy danh sách sản phẩm phân trang (mặc định page=0, size=10). Hỗ trợ tìm kiếm theo tham số name.
GET /api/products/{id} Public Lấy chi tiết sản phẩm theo mã ID. Trả về 404 Not Found nếu không tồn tại.
POST /api/products ADMIN Tạo sản phẩm mới. Yêu cầu giá và kho >= 0.
PUT /api/products/{id} ADMIN Cập nhật thông tin sản phẩm. Trả về thông tin sản phẩm sau khi sửa.
DELETE /api/products/{id} ADMIN Xóa sản phẩm khỏi hệ thống. Trả về 244 No Content.

Nhóm API Đặt hàng (Orders)

Method Endpoint Quyền hạn Mô tả
POST /api/orders USER Đặt hàng đồng bộ. Trả về 201 Created và trừ DB trực tiếp.
POST /api/orders/async USER Đặt hàng bất đồng bộ tốc độ cao. Trả về 202 Accepted và đẩy queue RabbitMQ.
GET /api/orders USER / ADMIN Xem lịch sử đặt hàng của tài khoản hiện tại.

7. Error Spec

7. Định dạng phản hồi lỗi chuẩn (Global Error Responses)

Khi hệ thống gặp lỗi nghiệp vụ, bảo mật hoặc validation đầu vào, mã nguồn sẽ bắt lỗi tại GlobalExceptionHandler và trả về JSON theo chuẩn định dạng nhất quán dưới đây:

Lỗi Validation dữ liệu đầu vào (Ví dụ: Thiếu tên sản phẩm hoặc giá trị âm)

{
    "timestamp": "2026-06-02T10:46:12.345",
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": {
        "name": "Product name is required",
        "price": "Price must be greater than or equal to 0"
    }
}

Lỗi Nghiệp vụ (Ví dụ: Không đủ tiền trong ví để thanh toán)

{
    "timestamp": "2026-06-02T10:46:30.123",
    "status": 400,
    "error": "Bad Request",
    "message": "Insufficient balance. Order total: 1200.00, your balance: 400.00"
}

Lỗi Phân Quyền (Ví dụ: Tài khoản USER gọi API ADMIN)

{
    "timestamp": "2026-06-02T10:46:45.987",
    "status": 403,
    "error": "Forbidden",
    "message": "Access denied: you do not have permission to access this resource"
}
