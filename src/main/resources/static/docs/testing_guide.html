Tài Liệu Hướng Dẫn Kiểm Thử
(Tester Manual Guide)

Tài liệu hướng dẫn chi tiết dành cho Tester để thực hiện bắn API qua Postman và kiểm chứng các kịch bản hoạt động của hệ thống.

1. Chuẩn bị môi trường

Thông tin môi trường & Khởi chạy

Địa chỉ cổng API mặc định của máy chủ Backend cục bộ: http://localhost:8080

Trước khi kiểm thử, Tester yêu cầu lập trình viên chạy cụm Docker Compose: docker compose up --build -d.

Tài khoản mẫu được tạo sẵn (Default Accounts)

Username Password Role Số dư ví mặc định Mục đích kiểm thử
user password ROLE_USER 1000.00 USD Kiểm thử mua hàng, kiểm tra số dư và phân quyền.
admin adminpassword ROLE_ADMIN 0.00 USD Kiểm thử CRUD sản phẩm, phân quyền quản trị viên.

Kết nối trực tiếp Cơ sở dữ liệu (Direct Database Connection)

Tester có thể sử dụng các công cụ quản lý DB (như DBeaver, pgAdmin, hoặc Navicat) để kết nối trực tiếp vào cơ sở dữ liệu PostgreSQL local nhằm kiểm tra dữ liệu thực tế lưu trữ trong bảng:

Thông số Giá trị kết nối Lưu ý kiểm thử DB
Database Type PostgreSQL Chọn Driver PostgreSQL trong công cụ DB client.
Host localhost (Hoặc địa chỉ IP máy của Developer nếu kết nối từ thiết bị mạng LAN khác).
Port 5432 Cổng tiêu chuẩn đã expose qua Docker Compose.
Database Name shopee_mini Cơ sở dữ liệu chính của dự án.
Username sa_user Tài khoản truy cập DB.
Password sa_password Mật khẩu truy cập DB.
Mục tiêu xác thực DB cho Tester:
  • Bảng users: Xác minh mật khẩu đăng ký được băm (hash) dạng BCrypt và kiểm tra số dư ví balance bị trừ chính xác sau khi đặt hàng.
  • Bảng products: Kiểm tra lượng tồn kho stock cập nhật tức thời khi mua thành công, và đảm bảo hoàn toàn không bị thay đổi nếu đơn hàng bị lỗi (kiểm tra rollback).
  • Bảng orders & order_items: Xác minh trạng thái đơn hàng và giá trị tiền của đơn hàng được chốt chính xác tại thời điểm mua.
Kịch bản kiểm thử 1

Quản Lý Sản Phẩm (Product Management CRUD)

Nhóm kịch bản kiểm thử các tính năng truy vấn, tạo mới, chỉnh sửa và xóa sản phẩm không cần và có cần vai trò quản trị.

Kịch bản 1.1: Lấy danh sách tất cả sản phẩm (Public API)

  • Method: GET
  • URL: http://localhost:8080/api/products
  • Kết quả mong đợi: Trả về mã HTTP 200 OK kèm JSON danh sách sản phẩm phân trang.

Kịch bản 1.2: Lấy chi tiết sản phẩm và Xem sản phẩm lỗi

  • Đúng cú pháp: GET http://localhost:8080/api/products/1 (200 OK)
  • Không tồn tại: GET http://localhost:8080/api/products/999 (404 Not Found)

Kịch bản 1.3: Biên dữ liệu lỗi khi thêm sản phẩm (Validation)

Đăng nhập tài khoản admin để lấy Token, dán vào cấu hình Authorization. Thực hiện thêm mới sản phẩm sai định dạng:

  • Method: POST
  • URL: http://localhost:8080/api/products
  • Body (JSON): 
    {
    "name": "",
    "price": -120.00,
    "stock": -5
}
  • Kết quả mong đợi (400 Bad Request): 
    {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": {
        "name": "Product name is required",
        "price": "Price must be greater than or equal to 0",
        "stock": "Stock must be greater than or equal to 0"
    }
}

Xác Thực Đăng Ký & Đăng Nhập (JWT Authentication)

Nhóm kịch bản kiểm tra an toàn bảo mật khi tạo tài khoản và sinh chuỗi Token JWT xác thực.

Kịch bản 2.1: Đăng ký thành công tài khoản kiểm thử

  • Method: POST
  • URL: http://localhost:8080/api/auth/register
  • Body (JSON) hỗ trợ nạp tiền ví để test: 
    {
    "username": "tester_wife",
    "password": "mypassword",
    "email": "wife@tester.com",
    "role": "ROLE_USER",
    "initialBalance": 2000.00
}
  • Kết quả mong đợi (201 Created): Trả về thông tin User. Mật khẩu password được lưu trữ dưới dạng mã hóa BCrypt băm an toàn trong PostgreSQL.

Kịch bản 2.2: Đăng ký trùng tên và Đăng nhập sai mật khẩu

  • Trùng tên: Gửi lại body trên lần nữa. Hệ thống trả về 400 Bad Request thông báo: "Username is already taken".
  • Sai mật khẩu: Đăng nhập với mật khẩu không đúng. Hệ thống trả về 401 Unauthorized với tin nhắn: "Invalid username or password".
Mẹo phân tích Token JWT:
Sau khi đăng nhập thành công qua POST /api/auth/login, Tester copy chuỗi token nhận được đem dán vào trang web jwt.io để phân tích xem có đúng thông tin role và sub (tên đăng nhập) không.

Phân Quyền Theo Vai Trò (Role-Based Access Control)

Nhóm kịch bản xác minh hệ thống ngăn chặn hành vi xâm nhập trái phép hoặc sai vai trò truy cập API.

Kịch bản 3.1: Gọi API được bảo vệ không truyền Authorization Header

  • API: GET http://localhost:8080/api/orders (Lấy lịch sử đơn hàng của tài khoản).
  • Kết quả mong đợi (403 Forbidden hoặc 401 Unauthorized): Bị chặn ngay từ tầng lọc bảo mật Spring Security do thiếu token hợp lệ.

Kịch bản 3.2: Dùng tài khoản USER gọi API của ADMIN (IDIOT Test)

  • Bước 1: Đăng nhập tài khoản user để lấy Token ROLE_USER.
  • Bước 2: Dán Token vào cấu hình Bearer Token.
  • Bước 3: Gửi request xóa sản phẩm: DELETE http://localhost:8080/api/products/1.
  • Kết quả mong đợi (403 Forbidden): Trả về thông báo từ chối truy cập chuẩn JSON: 
    {
    "status": 403,
    "error": "Forbidden",
    "message": "Access denied: you do not have permission to access this resource"
}
Kịch bản kiểm thử 4

Quy Trình Đặt Hàng & Giao Dịch (@Transactional Rollback)

Nhóm kịch bản quan trọng nhất kiểm chứng tính chính xác của ví tiền, kho bãi và tính toàn vẹn dữ liệu trong cơ sở dữ liệu.

Kịch bản 4.1: Đặt đơn hàng thành công và Trừ tiền ví

Đăng nhập tài khoản user (Ví có 1000.00 USD) và đặt mua 1 chiếc iPad Air (ID: 2, Giá 600.00 USD):

  • Method: POST
  • URL: http://localhost:8080/api/orders
  • Body (JSON): 
    {
    "items": [
        {
            "productId": 2,
            "quantity": 1
        }
    ]
}
  • Kết quả mong đợi (201 Created): * Trả về thông tin hóa đơn với trạng thái PAID. * Ví tiền còn lại remainingBalance chính xác bằng 400.00 USD. * Gửi truy vấn GET /api/products/2, tồn kho sản phẩm giảm từ 5 xuống 4.

Kịch bản 4.2: Đặt đơn hàng vượt số dư ví (Ví còn 400.00, mua iPhone 1200.00)

  • Gửi yêu cầu mua iPhone 15 Pro (ID: 1, Số lượng: 1).
  • Kết quả mong đợi (400 Bad Request): Trả về thông tin: "Insufficient balance. Order total: 1200.00, your balance: 400.00".

Kịch bản 4.3: Đặt đơn hàng vượt quá tồn kho (Kho còn 4, mua 5)

  • Gửi yêu cầu mua sản phẩm iPad Air (ID: 2) với số lượng là 5.
  • Kết quả mong đợi (400 Bad Request): Trả về thông tin: "Product 'iPad Air' is out of stock. Available stock: 4, requested: 5".

Kịch bản 4.4: Kiểm thử rollback dữ liệu giao dịch cực hạn

Đăng nhập tài khoản tester_wife (Ví có 2000.00 USD). Đơn hàng gồm 2 sản phẩm: Sản phẩm 3 hợp lệ (mua 2 AirPods Pro) và Sản phẩm 2 lỗi kho (mua 5 iPad Air trong khi kho chỉ còn 4 chiếc):

  • Body (JSON): 
    {
    "items": [
        {
            "productId": 3,
            "quantity": 2
        },
        {
            "productId": 2,
            "quantity": 5
        }
    ]
}
  • Bước 1: Bắn API đặt đơn hàng này. Kết quả mong đợi là hệ thống trả về lỗi 400 Bad Request báo sản phẩm 2 (iPad Air) hết hàng.
  • Bước 2 (Xác minh tính nguyên tử): Gọi API công khai GET http://localhost:8080/api/products/3.
    KẾT QUẢ ĐẠT YÊU CẦU Tồn kho của sản phẩm 3 (AirPods Pro) phải giữ nguyên là 20, hoàn toàn không bị giảm đi 2 chiếc! Điều này chứng minh toàn bộ các cập nhật của các mặt hàng trước đó đã được rollback thành công hoàn tất. Số dư ví người dùng vẫn giữ nguyên là 2000.00 USD.
